## 2. 计划结论

本轮不能把任务理解为“给两个占位页换皮”。目标是将已确认的视觉方向落成可恢复、可测试、连接真实业务契约的管理端与学习端，并且不让前端接管后端业务状态。

推荐实施顺序为十个阶段（编号 0 至 9）：

```text
阶段 0：只读核对鉴权、会话、任务 DTO、业务生命周期和读取契约，并拆出独立前置任务
-> 阶段 1：前端测试底座、设计 token 和基础组件
-> 阶段 2：两端应用壳、路由、API client 和状态边界
-> 阶段 3：最薄真实双端 walking skeleton
-> 阶段 4：学习码、课程首页和开始/继续课时
-> 阶段 5：TaskShell 与 S0-S5 六类任务逐个 TDD
-> 阶段 6：反馈、断网重试、错词回炉、刷新恢复和课后报告
-> 阶段 7：完整管理端“导入—构建—审阅—发布—建课”
-> 阶段 8：隔离真机预览、响应式、无障碍、视觉回归和儿童观察测试
-> 阶段 9：隔离 E2E、预览部署、全量门禁、生产 smoke 与回滚准备
```

阶段 0 不是可省略的准备工作，但它只做只读核对、决策收敛和前置任务拆分，不授权修改后端、数据库或 Cloudflare 配置。当前真实接口和安全边界不足以承载完整设计稿；所需后端与安全改动必须形成独立任务并另获用户编码授权。前置任务未完成时，不得用硬编码数据把对应页面标记为业务完成。

本计划初稿只授权产出实施方案；用户随后已明确授权按本计划执行。当前执行结果、验证证据和生产阻断以第 1 节所列实施验收与冲突报告为准。

## 3. 本计划采用的前提

### 3.1 已采用的设计前提

依据用户本轮“好的，然后根据这份设计计划”的指令，本计划按以下方向继续：

1. 采用青绿色主色、珊瑚色错误状态、阳光黄色重点状态和完整中性色阶。
2. 第一版采用抽象 `Aa` 词块，不增加具象品牌角色和完整插画系统。
3. 学习端优先服务 7 至 12 岁儿童，手机和平板优先，桌面可用。
4. 管理端采用桌面优先、克制、高密度、可扫描的工作台，不复制学习端的卡通化密度。
5. 学习端一屏一个主要任务；管理端围绕“导入、构建、审阅、发布”流水线组织页面。
6. 生产实现继续使用 Vue 3、TypeScript、scoped CSS 和全局 CSS 变量，不引入 Tailwind、UnoCSS 或完整 UI 组件库。

正式品牌名、具象角色、音频资产和插画体系仍不属于本轮范围。

### 3.2 技术前提

- 继续使用现有 Vue 3、Vue Router、Pinia、Zod、Vite、Vitest 和 Playwright。
- 前端只通过 `/api/admin/*` 或 `/api/app/*` 访问服务端，不访问 D1。
- 后端返回的数据是权威状态；Pinia 只保存当前交互状态和可从服务端恢复的缓存。
- 课时、阶段、评分、升级、降级、解锁和下次复习课时仍由后端决定。
- 所有任务由 `lessonTask` 快照驱动，前端不得重新读取练习项目拼题。
- 第一版不通过 `localStorage` 或 `sessionStorage` 保存权威课程状态、管理员凭证或学习者凭证。

### 3.3 待确认实施提案

- 词表导入提案为 UTF-8 CSV，由浏览器本地解析为结构化词数组后提交；文件本身不上传、不存入 D1 或 R2。该提案在确认前不是既定范围。

- 图标按设计稿使用 Lucide Vue；若实施时新增依赖，只允许新增实际使用且经过体积、许可证和锁文件复核的最小依赖。
- 自动化视觉截图只在页面功能、字体回退和动效行为稳定且用户完成首轮视觉确认后建立基线，避免把未定稿界面锁死。

## 5. 阶段 0 必须消解的阻断

### 5.1 管理员鉴权阻断

当前 `/api/admin/*` 只接受 `x-admin-token`，而 `/admin` 静态 SPA 入口仍可公开加载。浏览器不能安全持有 `ADMIN_API_TOKEN`。

禁止方案：

- 禁止把 `ADMIN_API_TOKEN` 写入 `VITE_*` 环境变量。
- 禁止把令牌打进静态 bundle。
- 禁止把令牌保存到 localStorage、sessionStorage、Pinia 持久化或普通 Cookie。
- 禁止为了联调临时移除管理 API 鉴权。

进入管理端真实数据接入前，必须选择并验证以下一种方案：

1. 推荐：Cloudflare Access 同时保护 `/admin*` 和 `/api/admin/*`，Worker 校验可验证的 Access 身份。
2. 备选：应用层管理员登录，服务端建立 Secure、HttpOnly、SameSite 会话 Cookie。

Access 方案还必须确认可用的自定义域名或已验证的等价路径保护条件。Cookie 方案还必须冻结会话有效期、注销、轮换、Origin 校验、CSRF 防护和失败错误码；不能只设置 Cookie 属性就认定写请求安全。

若两者都未完成，只允许实现静态管理端壳和隔离 fixture，不得接生产管理数据，也不得宣称管理端通过生产安全验收。

### 5.2 学习者会话与资源归属阻断

当前学习码接口返回 learner/course，后续请求主要依赖 `courseId`、`sessionId` 和 `taskId`。服务端尚未建立浏览器可验证的学习者会话边界。

阶段 0 必须明确：

- 学习码兑换成功后如何建立服务端会话。
- app 请求如何证明当前 learner 可以访问目标 course。
- lesson 和 task 如何验证属于当前 course。
- 会话失效、学习码错误和跨课程访问分别返回什么稳定错误码。
- 会话有效期、注销/清理、学习码轮换和 Cookie 写请求的 Origin/CSRF 规则。

推荐使用 Secure、HttpOnly、SameSite Cookie 或等价的服务端可验证会话。仅把 learner/course 放进 Pinia 不能满足权限和刷新恢复要求。

### 5.3 任务 DTO 阻断

当前 `LessonTaskView` 的 `stage`、`taskType` 是宽泛字符串，`prompt` 是 `unknown`。这会迫使 renderer 使用类型断言并猜测 JSON 结构，违反共享 schema 和显式契约规则。

进入任何任务 renderer 前，必须完成：

- `WordStage` 和 `TaskType` 使用既有枚举联合。
- 六类 prompt 建立以 `taskType` 为判别字段的联合类型和 Zod schema。
- API 响应在 client 边界运行时校验。
- renderer 使用穷尽分支；新增 task type 时编译和测试必须失败，直到补齐 renderer。
- app 响应不返回权威答案字段；正确答案只在提交后的反馈 DTO 中按业务规则返回。

必须单独确认的交互契约：

- S0 `recognize_meaning` 是“认识/不认识”、单纯确认，还是其他评分行为。
- S5 `sentence_output` 是服务端自动评分、人工/自评 0-3，还是两段式反馈。
- S4 `sentence_build` 提交有序词块数组还是规范化字符串。
- 答错后新增的回炉任务通过 answer 响应返回，还是通过读取当前 lesson 状态取得。

前端不能自行发明这些评分语义。

当前内容数据也不能直接作为生产题目契约：S2 干扰项仍是模板后缀，S4 词块按正确顺序生成，S0 prompt 同时包含单词、释义和例句。阶段 0 必须把它们分别判定为“符合题型意图”或“独立内容构建缺口”；前端不得通过隐藏字段、随机打乱或本地造干扰项掩盖服务端内容问题。

### 5.4 页面读取、恢复与业务生命周期阻断

为避免页面刷新后丢失状态，至少需要以下服务端读取能力；路由名称可在契约评审中调整，但能力不可缺失：

| 页面能力 | 当前状态 | 最小契约要求 |
| --- | --- | --- |
| 词库版本列表 | 缺失 | 返回 source、versionNo、status、wordCount、groupCount，以及现有 `createdAt` / `publishedAt`；不为展示默认新增 updatedAt |
| 版本详情 | 缺失 | 返回草稿/已发布状态、导入摘要、构建摘要和允许操作 |
| 同 source 新版本 | 缺失；当前 import 每次新建 source 且 versionNo 固定 1 | 按上位计划实现“重复导入在同一 source 下创建下一 draft version”、versionNo 递增和不可变测试；若要缩减，必须先修订上位计划 |
| 覆盖率矩阵 | 只有 build 返回摘要 | 可重新读取按 word/stage 的覆盖状态和发布阻断项 |
| 练习详情、编辑、批准 | 缺失；当前 builder 直接生成 approved item | 按上位计划实现生成 draft、管理员查看/编辑/批准/禁用、只有 approved 计入发布覆盖；若要保留自动批准，必须先修订上位计划 |
| 学习者与课程查看 | 只有创建返回值 | 可恢复已创建课程、学习码展示状态和最小课程状态视图 |
| 课程首页摘要 | 缺失 | 返回 course、当前课号、新词/复习词摘要、开始/继续状态，以及前一课/当前课/下一课的最小 `lessonPath` 节点和 completed/current/locked 状态；前端不得推导路径状态。预计时长只有服务端提供明确派生规则时才展示，否则本轮移除 |
| 当前课恢复 | start 可复用但语义不完整 | 返回同一 session、完整 task 快照、完成状态和新增回炉任务 |
| 课后报告 | 缺失 | 返回完成数、正确率、需再练词、进步词和下一课入口数据 |

不允许用硬编码数字、HTML 审阅稿示例数据或 localStorage 替代这些契约。

### 5.5 错词回炉与完成规则冲突

本计划初稿依据的旧规则是“同词第一次再次出现前隔 5 至 8 道题且同课完成”。五词连续全错证明该组合无解，历史反例与当时的实现审计保留在五词冲突报告中，不得删除或改写成已通过。

2026-07-14，Solazhu 已通过 `PLAN_0714_错词回流间隔与单词重复上限优化_v1.md` 确认 G-W v2 主框架：

- 答错后第一次再次完成相同 `wordId` 前完整间隔 3 至 6 道实际完成题；skipped task 不计间隔。
- 每词每课 `primary + bridge + reflux` 总 task 上限为 3，pending 和 skipped task 也占预算。
- 所有可评分 role 答错统一进入服务端队列策略；达到上限写 `deferred_cap`，精确可行性判断证明无合法排程时写 `deferred_capacity`。
- defer 后不再追加本课 required task，该词由 StageEngine 收紧为下一课到期。
- 结课仍要求至少完成 80% primary 和全部已生成 required task；deferred outcome 不是 pending obligation，不永久阻断本课。

原始 5 道任务中前 4 道答对、最后 1 道答错的 `5 / 6 = 83.3%` 反例仍必须覆盖：若已生成 required 回流，则 complete 必须失败直到它完成；若权威 outcome 已明确 defer，则不得制造不存在的 pending obligation。五词连续全错还必须恰好 15 题、每词 3 次且无第 4 次。

2026-07-14，v2 已完成纯队列、Service、SQLite-D1、Worker API、恢复/幂等和浏览器闭环的本地实现与内部验收。目标 D1 started v1 审计、远端 migration 和隔离 preview 尚未执行，因此只关闭本地 G-W 逻辑阻断，生产仍保持 NO-GO。

### 5.6 错误契约阻断

当前多数 service 错误最终退化为 `409 conflict` 和一段 message，无法可靠驱动字段错误、发布阻断、已发布只读、权限失败和网络重试界面。

阶段 0 必须冻结稳定错误结构：

```text
code: 稳定机器码
message: 开发与兜底信息
details: 可选的字段错误、阻断项或资源上下文
```

前端只按 `code` 和经过 schema 校验的 `details` 分支，不解析英文 message 判断业务状态。

### 5.7 阶段 0 通过标准

阶段 0 本身不实现后端与安全改动；它只核对现状、冻结细节、列出验收用例并拆分独立任务。阶段 0 的只读交付完成后即可进入不接真实业务数据的阶段 1 和阶段 2，不要求无关前置任务全部先关闭。

真实数据切片按依赖分别过门，不采用一个全局总门禁：

| 门禁 | 必须证据 | 最晚关闭时间 |
| --- | --- | --- |
| G-A 管理员身份 | Access/HttpOnly 方案、写请求安全、无静态 secret | 阶段 3 首次 admin 真实请求前 |
| G-L 学习者会话 | learner session、资源归属、会话生命周期 | 阶段 3 首次 app 真实请求前 |
| G-S 最小任务契约 | S0 判别联合、提交语义、答案不泄露 | 阶段 3 S0 渲染前 |
| G-V 版本与审核 | 同 source 新 draft、versionNo 递增、draft → approved → disabled、发布不可变 | 阶段 3 publish 前 |
| G-R 读取恢复 | walking skeleton 所需 source/course/session 读取；其他读取按页面补齐 | 阶段 3 及各对应页面前 |
| G-T 六类任务 | 对应 task prompt/answer schema 和内容质量门 | 阶段 5 每个 renderer 前逐个关闭 |
| G-W 回流与有限结课 | 3-6 道实际间隔、每词总 task 上限 3、cap/capacity defer、五词全错 15 题、最后一题答错反例 | 阶段 6 回流/complete 前 |
| G-E 稳定错误 | 当前切片所需 code/details | 每个真实交互切片前 |
| G-P preview | 独立 Worker/D1、fail-closed 身份校验 | 阶段 8 真机观察前 |

任一门禁未满足，只停止依赖它的真实数据切片；不得用 fixture 把该切片标记为业务完成。原有服务、API 和部署基线在每个独立前置任务后都必须继续通过。

## 6. 建设目标与非目标

### 6.1 建设目标

#### 管理端

- 管理员能导入、预览、校验一份词表。
- 管理员能执行构建并查看真实覆盖率和阻断项。
- 管理员能查看、编辑、批准和禁用草稿练习项目，只有 approved 项目计入发布覆盖。
- 覆盖率不达标时，发布按钮禁用且列出可行动原因。
- 版本发布后页面只读，并提供在同一 source 下创建下一 draft version 的入口。
- 管理员能为学习者创建课程、取得学习码并查看最小课程状态。

#### 学习端

- 学习者能输入学习码进入自己的课程。
- 课程首页只突出第 N 课和开始/继续动作。
- 学习者能完成 S0-S5 的真实任务。
- 正确、错误和网络失败均有明确、可继续的反馈。
- 刷新或中途退出后能从服务端恢复同一未完成课时。
- 错词回炉任务不被漏掉，完成条件由后端决定。
- 完成课时后显示儿童可理解的报告和下一课入口。

#### 工程质量

- 前后端共享 API schema，Vue 组件不直接拼 URL 或裸 fetch。
- 页面、feature、store/composable、API client 和基础 UI 边界清楚。
- 关键行为先有失败测试，再写最小实现。
- 指定视口、键盘、减少动效、错误恢复和真实 API 闭环通过验收。

### 6.2 非目标

- 不增加 AI 出题、AI 评分、语音识别、R2 资源上传、支付、多租户或 SaaS 权限矩阵。
- 不增加自然日 streak、红心、生命值、排行榜、货币、商店或社交功能。
- 不建设正式品牌命名、具象角色、完整插画和音频资产系统。
- 不新增通知中心、后台设置、导出中心或视觉稿中的死导航。
- 不重写后端调度、StageEngine 或 D1 repository；本计划只列出前端所依赖的最小契约补齐。
- 不引入完整组件库、Storybook、通用低代码表单或通用 feature flag 平台。
- 不把视觉审阅 HTML 直接复制成生产 Vue 页面。

## 7. 页面与路由范围

路由在实施时从 `src/main.ts` 移到独立 router 模块，并按端隔离。

### 7.1 管理端

| 路由方向 | 页面职责 | 必须状态 |
| --- | --- | --- |
| `/admin` | 概览或重定向至词库版本 | loading、empty、unauthorized、error |
| `/admin/source-versions` | 词库与版本列表、导入入口 | empty、normal、importing、import failed |
| `/admin/source-versions/:versionId` | 导入摘要、构建流水线、覆盖率、阻断项 | draft、building、incomplete、ready、published immutable |
| `/admin/source-versions/:versionId/exercises/:itemId` | 草稿练习详情、编辑、批准与禁用 | loading、validation error、saved、approved、disabled、published readonly |
| `/admin/courses` | 最小学习者/课程创建、学习码展示与课程状态 | empty、created、active、paused、completed、conflict、error |

如果某个导航目标在当前迭代没有真实页面和数据，不展示该导航项。

### 7.2 学习端

| 路由方向 | 页面职责 | 必须状态 |
| --- | --- | --- |
| `/app` | 学习码进入或恢复已有服务端会话 | idle、submitting、invalid code、session expired |
| `/app/course` | 当前课程和第 N 课摘要 | loading、start、continue、course unavailable |
| `/app/lesson/:sessionId` | 当前任务、提交、反馈和进度 | loading、answering、submitting、feedback、retry、completing |
| `/app/lesson/:sessionId/report` | 课后报告和下一课入口 | loading、complete、report unavailable |

具体 URL 可在路由设计时微调，但不得把 admin 与 learner 页面混入同一访问分区。

只创建当前阶段实际需要的目录和文件，不一次性建立空目录骨架。

第一版继续把 API DTO 和对应运行时 Zod schema 统一放在 `shared/api/`，从现有 `shared/api/schemas.ts` 按真实增长需要拆文件；不同时建立 `shared/schemas/`，避免两套事实来源。领域枚举继续放 `shared/domain/`。

### 8.2 API client 边界

- `httpClient.ts` 只负责请求、统一响应解析、运行时 schema 校验和错误标准化。只有路由切换或重复请求出现已验证的取消需求时，才增加取消能力。
- `adminApi.ts` 只能构造 `/api/admin/*` 请求。
- `learnerApi.ts` 只能构造 `/api/app/*` 请求。
- Vue Page、feature、store 和 renderer 不得出现裸 fetch 或 API base URL。
- app client 不得导入 admin client，也不得携带管理员凭证。
- 后端原始 message 不直接作为主要用户文案；前端根据稳定 code 映射可行动文案。

### 8.3 状态边界

Pinia 或 composable 只保存：

- 当前页面的 loading/error 状态。
- 当前 session/task 的可恢复展示缓存。
- 尚未成功提交的本地输入。
- 当前焦点、选中项、展开项等纯交互状态。

Pinia 不保存：

- 权威 `currentLessonNo`。
- 权威 stage、mastery、review log 或发布状态。
- 管理员 secret。
- 可绕过服务端恢复和权限检查的长期业务快照。

刷新后必须重新从服务端取得会话、课时任务和版本状态。

### 8.4 视觉实现边界

- 颜色、字号、间距、圆角、阴影、层级和动效时长统一进入 CSS token。
- 学习端和管理端共享 token，不共享布局密度。
- 学习端高频目标不小于 48×48 CSS px；管理端目标不小于 40×40 CSS px。
- 学习端答案选项最小高度 56px，学习端输入框高度 52px；48px 是通用触控底线，不能替代组件规格。
- 正确、错误、选中、禁用、发布状态均同时使用文字、图标/形状和颜色。
- 图标按钮必须有可访问名称；陌生图标提供可见 tooltip，关键操作不得依赖 hover 才能发现。
- 高对比/forced-colors 模式下保留边界、焦点和状态区分。
- 所有交互动效支持 `prefers-reduced-motion`；减少动效时不丢失状态信息。
- 字体优先使用系统回退或自托管资产，不依赖在中国大陆不稳定的公共字体 CDN。

## 9. 防错执行方法

### 9.1 每个纵向切片的固定循环

```text
确认该切片的数据契约与权限边界
-> 写一个可观察行为的失败测试
-> 写最小实现使测试通过
-> 补该行为的错误与恢复测试
-> 运行切片相关的 typecheck/lint/test
-> 在指定视口人工检查
-> 运行全量回归
-> 记录结果后才进入下一切片
```

禁止先铺完所有页面，再统一接 API 或补测试。

### 9.2 Definition of Ready

一个切片进入编码前必须具备：

1. 页面目标和单一主要行为明确。
2. 所需 API、schema、错误码和权限已经存在并通过契约测试。
3. 成功、空、加载、失败、重试和权限状态已列出。
4. 测试 fixture 使用真实 DTO，不复制视觉稿示例对象。
5. 该切片不要求前端计算调度、阶段或发布规则。
6. 对应视觉方向已经确认。

### 9.3 Definition of Done

一个切片只有同时满足以下条件才算完成：

1. 正常路径连接真实 API 或明确标注为纯视觉基础组件。
2. 失败路径不会清空已有输入、推进错误状态或伪造成功。
3. 刷新后能从服务端恢复，或明确属于无业务状态的静态页面。
4. 单元/组件/E2E 中与该切片相关的测试通过。
5. 键盘、焦点、可访问名称和状态消息通过检查。
6. 目标视口无横向溢出、遮挡和主操作竞争。
7. 全量 typecheck、lint、test、build 未回归。

## 10. 详细实施阶段

### 阶段 0：只读前置验收与独立任务拆分

目标：只读确认前端无法独立解决的高风险缺口，冻结必要产品决策，并把后端、安全和基础设施改动拆成需另获用户编码授权的独立任务。本阶段不修改 `server/`、`shared/`、`migrations/` 或 Cloudflare 配置。

任务：

1. 建立“页面—DTO—API—错误码—权限”矩阵。
2. 决定管理员 Access 或服务端会话方案及其外部依赖。
3. 决定学习码换取 learner session 的契约和资源归属规则。
4. 定义六种 task 判别联合、S0/S4/S5 提交语义和任务内容质量门。
5. 细化上位计划既定的 source/version 生命周期：新 source、同 source 下一 draft、重新导入和发布后不可变。
6. 按上位计划细化 draft → approved → disabled 人工审核流；若要缩减为自动批准，先提交上位计划修订而不是继续前端实施。
7. 列出页面所需的最小读取、恢复、报告和稳定错误契约。
8. 按 `PLAN_0714` 把“3 至 6 道实际间隔 + 每词总上限 3 + defer 后有限结课”拆成独立后端修复任务。
9. 为每个独立任务写明红测、最小改动范围、验收和回滚，并请求用户分别授权后再编码。
10. 运行现有全量门禁，记录编码前基线。

前置任务必须交回的验证证据：

- API 契约测试覆盖合法与非法 task prompt。
- 安全测试覆盖无 admin 身份、无 learner session、跨 course/session/task 访问。
- 响应测试确认未提交任务不包含答案。
- 重复 start 返回同一未完成 session 和同一 task 快照。
- answer 后可重新取得包含新增回炉任务的完整 session 状态。
- 前四题正确、最后一题错误时，已生成 required 回流完成前 complete 被拒绝；若权威结果为 defer，则不存在虚假的 pending obligation。
- 五词连续全错恰好 15 题、每词 3 次；所有实际安排回流的首次同词再现前完整间隔 3 至 6 道题。
- 同 source 新版本和练习项目状态流按上位计划通过 service/API 测试。

验收标准：阶段 0 的只读交付是决策和独立任务清单；真实数据接入门禁以第 5.7 节所列独立任务验证证据为准。任何安全、生命周期、回炉或任务语义未闭合，停止进入对应页面编码。

### 阶段 1：测试底座与设计基础

目标：在业务页面出现前，先让 Vue SFC、交互组件和目标视口具备真实测试门禁。

任务：

1. 使用 `eslint-plugin-vue` 让 ESLint 实际覆盖 Vue SFC，并更新对应 TypeScript lint project。
2. 保留现有 Node Vitest project 运行后端/API 测试；另建一个使用现有 Vue 编译插件、`@vue/test-utils` 和 `happy-dom` 的组件测试 project，不把后端测试全局切到 DOM 环境。
3. 让 `pnpm test` 聚合 Node 与组件 project，同时保留可单独运行的聚焦命令。
4. 建立 `test:e2e:ui` Playwright project，并增加学习端和管理端目标视口。
5. 在阶段 3 前建立最小 `test:e2e:stack` 载体：独占端口、`reuseExistingServer=false`、全新临时 D1 状态目录、suite 前 migration、suite 后销毁、受控测试身份注入，以及启动后的 health/环境/数据库身份检查。认证 setup 默认关闭 trace/HAR，并将临时 storage state 放入 suite 临时目录后销毁。
6. 建立稳定 fixture：固定业务数据、固定时间、稳定字体回退、关闭非必要动效。
7. 创建 CSS token、基础 reset、字体、焦点和 reduced-motion 规则。
8. 按真实复用需要实现最小基础组件：Button、Input、StatusBadge、Progress、Feedback、Loading、Empty、Error。
9. 接入 `@lucide/vue`；阶段 8 使用 `@axe-core/playwright`。除上述有明确测试或界面用途的依赖外，不引入完整 UI 框架或额外工具链。

测试方式：

- 组件测试覆盖事件、ARIA、disabled、loading、error、焦点状态和状态切换。
- Playwright 真实浏览器覆盖 hover、active、focus-visible、48/40px 触控底线、56px 答案选项、52px 学习输入框和 reduced-motion CSS。
- 键盘可操作 Button、Input 和主要状态组件。
- Playwright 与人工复核关键文本、控件和背景的对比度；DOM runtime 不承担真实尺寸或 CSS 视觉验收。

验收标准：

- Vue 文件纳入 lint。
- `pnpm test` 同时运行 Node 与 Vue 组件测试，且两者可独立聚焦运行。
- 320、375、768、1024、1280、1440 相关 Playwright 项目可运行。
- `pnpm test:e2e:stack` 能在不接触生产 D1、不复用旧服务的条件下完成最小 health 与数据写读 smoke，并自动销毁临时状态。
- token 与设计文档一致，页面未出现第二套散落色值和间距体系。

### 阶段 2：路由、两端应用壳、API client 与状态边界

目标：建立两端明确隔离且可承载后续纵向切片的最小应用框架。

任务：

1. 将 router 从 `src/main.ts` 移到独立模块，并使用页面级懒加载。
2. 实现 `AdminLayout`：稳定侧栏、顶部上下文、主内容和跳到主内容入口。
3. 实现 `LearnerLayout`：低干扰画布、窄列内容和安全区适配。
4. 实现统一 `httpClient`、`adminApi`、`learnerApi` 和错误模型。
5. 实现最小 store/composable，只保存可恢复交互状态。
6. 管理端 unauthorized 时不渲染业务壳和业务数据。
7. 未实现的导航项不展示，避免死入口。

测试方式：

- 路由测试覆盖 `/`、admin、app、未知路由和未认证 admin。
- API client 测试覆盖 success、validation、401、403、404、409、依赖失败、非法响应和网络异常。
- Playwright 只收集请求 URL、method 和 namespace 用于断言，不序列化 headers、cookies 或 body；验证 app 页面不访问 `/api/admin/*`，admin 页面不使用 `/api/app/*` 复用管理能力。
- 刷新页面后 store 从服务端恢复，不从 localStorage 恢复权威状态。

验收标准：

- 两端视觉属于同一品牌，但密度明显不同。
- 未认证访问 admin 不发送业务数据请求。
- 任何 Vue 组件中没有裸 fetch、管理员 secret 或 API URL 拼接。

### 阶段 3：最薄真实双端 walking skeleton

目标：在扩展完整页面前，用最少 UI 和真实 Worker/D1 验证一次 admin → learner 跨端闭环，尽早暴露鉴权、会话、schema、持久化和任务反馈问题。

进入条件：阶段 0 拆出的管理员鉴权、learner session、S0 DTO，以及本阶段涉及的读取恢复契约等独立前置任务已经分别获授权、实现并通过对应门禁；错词完成规则和回流门禁在阶段 6 前关闭。

任务：

1. 使用阶段 0 确认的最小导入交互提交最小结构化测试词，不引入硬编码“一键造数据”生产按钮。
2. 通过真实 admin API 执行 import 和 build；build 后项目保持 draft，使用最小真实列表批准本 fixture 的全部必需项目，重新达到发布覆盖后再 publish。
3. 创建 course，并展示每步服务端结果。
4. 在 app 端输入刚生成的学习码，建立真实 learner session。
5. start lesson 后用判别联合 schema 渲染一个 S0 task。
6. 按已确认 S0 语义提交一次答案/响应，展示服务端反馈并重新取得权威 lesson 状态。
7. 该切片只做完成闭环所需的最小布局；版本列表、覆盖率矩阵、完整路径和六类 renderer 留给后续阶段。

测试方式：

- Playwright 真实栈用例从 admin 导入开始，直到 app 成功提交一个 S0 task。
- 请求日志断言 admin/app namespace 不串用，浏览器不包含管理员静态 secret。
- 所有响应通过共享 Zod schema；fixture 不复制 HTML 审阅稿的版本号和统计数字。
- 刷新后重新取得相同 source version、course、session 和 task 快照。

验收标准：

- 首次真实双端闭环在扩展完整管理页面前通过。
- 任何步骤失败时页面保留可恢复上下文，不伪造下一步成功。
- walking skeleton 使用的临时交互在后续深化时被真实页面接管，不长期保留第二套流程。

### 阶段 4：学习码、课程首页与开始/继续课时

目标：在 walking skeleton 已验证集成后，完成设计稿要求的学习码与课程首页体验。

任务：

1. 实现学习码输入、提交、错误和会话失效状态。
2. 学习码格式以服务端契约为准；当前 10 位契约与设计文档“4 至 6 位”提示必须在阶段 0 统一。
3. 成功后由服务端会话恢复 course，不把 access code 当长期凭证保存。
4. 实现课程首页：前一课、当前课、下一课的连续路径，第 N 课、复习/新词摘要和一个开始/继续按钮。
5. 预计时长只有服务端提供明确派生规则时才展示；否则本轮移除，不在前端按任务数猜测。
6. 页面只显示服务端返回的课时编号，不读取当前日期推导节奏。
7. 开始或继续后进入服务端返回的 session。

测试方式：

- 空码、非法码、过期会话和网络失败。
- 正确码只进入当前 learner 的 course。
- 已有 started session 时显示“继续第 N 课”。
- 将浏览器系统时间前后调整五天，只验证前端显示不受浏览器日期影响；服务端“停学五天不推进”由可注入服务端时间的 StageEngine/CourseRuntime 测试证明。
- 375px 下主操作唯一且无需横向滚动。

验收标准：

- 当前课和唯一开始/继续主操作在目标移动首屏无需滚动即可看见；“3 秒识别”移至阶段 8 的目标儿童观察测试。
- 首页不显示自然日连续学习、阶段算法、排名、货币或内部掌握指标。
- 刷新后通过服务端会话恢复同一 course。

### 阶段 5：TaskShell 与六类任务逐个 TDD

目标：用一个稳定任务骨架承载六类任务，组件只渲染和收集答案，不解释业务评分。

#### 公共 TaskShell

任务：

- 顶部：退出、第 N 课和本课进度。
- 中部：一条短指令、任务内容和作答区。
- 底部：提交前“检查答案”，反馈后“继续”，网络失败时“重新提交”。
- 提交期间禁用重复操作，但不清空答案。
- feedback 使用 `aria-live` 或等价状态消息，不强制移动焦点到不可预期位置。

#### Renderer 顺序

每次只实现一个 renderer，完成测试和全量回归后再进入下一个：

1. `RecognizeMeaningTask`：S0 认识任务，行为以阶段 0 确认的评分契约为准。
2. `RecallWordTask`：S1 释义回忆单词。
3. `MultipleChoiceTask`：S2 整行可点击选项，支持键盘选择。
4. `FillBlankTask`：S3 短句填空，保留输入和明确错误。
5. `SentenceBuildTask`：S4 句子拼装，必须提供无需拖拽的点击排序或移动按钮路径。
6. `SentenceOutputTask`：S5 主动输出，评分与反馈以阶段 0 契约为准。

每个 renderer 的测试：

- 正确 prompt 可渲染。
- 非法 prompt 在 API client schema 层失败，不进入 renderer。
- 默认、选中、填写、提交中、正确、错误、禁用和焦点状态。
- Enter/Space 和 Tab 顺序符合控件语义。
- 只提交用户答案或明确响应，不提交前端计算的 stage/score/nextDueLessonNo。
- task 响应和页面 DOM 在提交前不泄露正确答案。
- 新 task type 未注册时通过 TypeScript 穷尽检查阻断构建。

验收标准：

- 六类 renderer 均通过组件测试和 Playwright 行为测试。
- 任务页始终只有一个竞争性的主要动作。
- 不同题型切换不改变顶部、反馈和底部动作的基本位置。
- S4 不依赖拖拽，S5 不由前端自行定义评分。

### 阶段 6：反馈、重试、回炉、恢复与报告

目标：关闭最容易造成数据错乱的失败与重复操作路径。

任务：

1. 提交成功后根据服务端反馈展示结果、正确答案和短提示。
2. 网络失败保留答案、当前 task 和索引；只有服务端成功后才推进。
3. 重试沿用同一 task，后端幂等保证不重复计数。
4. answer 后重新取得权威 session/task 状态，确保新增回炉任务进入队列。
5. 刷新后定位第一个 pending task；已完成 task 不重复作答。
6. 完成本课由服务端判断；前端只展示阻断原因或成功结果。
7. 课后报告使用独立 DTO，不在前端从 task 列表推算正确率或掌握度。
8. 下一课入口使用服务端返回的最新 course 状态。

测试方式：

- 第一次提交模拟断网，答案和任务位置不变；恢复后重试成功且只记一次。
- 双击提交只产生一次有效 review log。
- 双击 complete 只推进一次 lesson。
- 任意位置答错后，若服务端实际安排回流，首次同词再现前完整间隔 3 至 6 道实际完成题；达到每词总 task 上限 3 或精确证明容量不足时，服务端持久化 defer，前端只渲染权威结果。
- 前四题正确、最后一题错误时，新增 required 回流可见，`5 / 6 = 83.3%` 仍不能完成课时，直到该 required task 完成；若该错答已明确 defer，则不会出现伪 pending。
- 五词连续全错恰好 15 题、每词出现 3 次且无第 4 次；第三次仍错进入下一课“需继续练习”。
- 刷新前后 session ID、task ID、顺序、prompt 快照和完成状态一致。
- 发布同 source 下一版本后，已开始课程和已生成 task 不发生变化。
- 报告不显示 `easeFactor`、内部 stage 计数或调度公式。

验收标准：

- 网络错误不会被记成答错。
- 前端不会在失败或超时后乐观推进课时。
- 未完成退出和浏览器刷新均能恢复。
- 报告能回答“完成了什么、哪里更稳、哪里要再练、下一步是什么”。

### 阶段 7：完整管理端内容构建闭环

目标：在学习端核心闭环稳定后，把 walking skeleton 深化为设计稿要求的“导入—构建—审阅—发布—创建课程”工作台，每一步都以服务端状态为准。

#### 切片 7A：词库列表、导入与预览

任务：

1. 实现版本列表的 loading、empty、normal 和 error。
2. 使用阶段 0 已确认的导入格式、字段命名、解析边界和依赖方案，不在本阶段重新决定 CSV 规则。
3. 展示总词数、空字段、重复词和具体行号。
4. 分组数优先使用服务端 preview/import 结果；若确需提交前展示，只能消费共享 `GROUP_SIZE` 单一事实源，不在前端硬编码第二份“每 5 词”规则。
5. 只有预览通过才允许提交导入；前端预校验只改善体验，服务端仍做权威校验。
6. 导入失败保留可修正输入和预览，不显示伪成功版本。

测试与验收：

- 20 个有效词由服务端确认 4 组，22 个有效词确认 5 组。
- 空 word、空 meaning、重复词和确认格式中的非法文件/表头有字段级错误。
- 导入成功后重新读取服务端版本，不只使用 POST 返回值留在本地。
- 同 source 下一版本、重新导入和新建 source 的 UI 语义与阶段 0 决策一致。

#### 切片 7B：构建与覆盖率

任务：

1. 显示导入、构建、审阅、发布四步状态；build 只生成 draft 项目，审阅阶段承担编辑、批准和禁用。
2. 执行构建时显示稳定状态和明确文案，不用无限转圈替代进度。
3. 展示真实总词数、分组数、练习数、覆盖率和按 word/stage 的矩阵。
4. “只看缺口”由真实覆盖 DTO 驱动。
5. 构建失败显示失败对象、保留现有草稿并提供重试。

测试与验收：

- 20 个完整词在 build 后项目保持 draft、发布仍被阻断；批准全部必需项目并重新读取后才达到 100%。
- 缺例句时，当前真实规则下 S3、S4、S5 缺口可见；不得硬编码审阅稿中的“只缺 4 个 S4”。
- 构建请求重复触发时按钮受控，服务端状态和结果不重复污染。

#### 切片 7C：练习详情与审核

任务：

1. 从覆盖率缺口进入具体词、阶段和题型详情。
2. 表单按 task schema 编辑，不直接编辑未校验 JSON。
3. 显示字段级错误、保存中、保存成功和冲突。
4. 草稿可编辑；已发布版本直接移除编辑入口并显示只读原因。
5. 批量操作只在真实批量 API 和明确选择状态存在时出现。

测试与验收：

- 修改后重新读取覆盖率，缺口真实减少。
- 已发布版本通过 UI 无编辑入口，直接调用写 API 也被服务端拒绝。
- 页面不通过颜色单独表达 missing/approved/disabled。

#### 切片 7D：发布、下一版本、学习者与课程

任务：

1. 覆盖率未达标时禁用发布并列出阻断项。
2. 发布确认显示版本号、词数、分组数、覆盖状态和不可变影响。
3. 发布成功后切换为服务端返回的只读状态。
4. 已发布版本提供“创建下一草稿版本”，并由服务端在同一 source 下递增 versionNo。
5. 使用已发布版本创建学习者与课程，展示学习码和 active/paused/completed 最小状态。
6. 学习码只按业务需要展示，不写日志或分析事件。

测试与验收：

- 100% 覆盖前 UI 不允许发布，直接 API 仍拒绝。
- 重复发布不生成不同成功状态。
- course 只能绑定 published source version。
- 发布后重新加载仍是只读状态；不能通过刷新恢复编辑入口。
- 创建同 source 下一版本后，旧 course 和既有 lesson task 继续绑定旧快照。

### 阶段 8：隔离真机预览、响应式、无障碍、视觉回归与儿童观察

目标：在真实业务闭环稳定后完成跨视口、输入方式和目标用户验证。

#### 真机观察环境

儿童观察开始前必须先关闭 G-P：另行授权并建立受限 preview Worker/D1，或使用可被测试手机/平板访问的受控局域网服务加全新本地 D1。两种方式都必须使用隔离测试数据、明确环境标识和清理步骤，禁止连接 `eng-learn-prod`，并先在至少一台真实手机和一台真实平板完成 smoke。

首选受保护 preview，因为它更接近真实网络、Cookie 和 Cloudflare 身份边界。若使用局域网方案，报告必须明确它未验证远端 Access/preview 行为，阶段 9 仍需补远端 preview 验收。

#### 自动化视口

学习端：

- 320×568
- 375×812
- 768×1024
- 1280×800

管理端：

- 1024×768
- 1280×800
- 1440×900

#### 自动化与人工检查

- 页面级横向溢出。
- 长中文、长英文和浏览器 200% 缩放。
- 移动端安全区和软键盘遮挡。
- 键盘完成学习码、选择、填空、拼装、提交和继续。
- 可见 focus、跳到主内容、标题层级、按钮/链接语义和 `lang`。
- 正确、错误、选中、禁用和发布状态在灰度下仍可区分。

- `prefers-reduced-motion` 下无状态丢失。

视觉截图规则：

- 使用固定 fixture、固定时间、稳定字体和关闭非必要动效。
- 先由用户确认首轮生产页面视觉，再建立 screenshot baseline。
- 截图差异只定位视觉回归，不替代业务和无障碍断言。

儿童观察测试：

- 这是需要真实参与者的外部门禁，提案为至少 3 名 7 至 12 岁儿童；招募和执行由 Solazhu 负责，并在监护人知情同意后进行。
- 默认不录音、不录像、不保存姓名；只记录匿名编号、年龄段、任务结果和观察笔记。
- 每人完成同一份 5 至 10 题脚本，覆盖选择、输入、S4 非拖拽拼装、错误反馈和一次模拟网络重试。
- 页面稳定呈现后，由主持人只说“请开始”，从该时点计时；不先讲解操作。至少 2/3 参与者应在 3 秒内无提示找到当前主操作。

验收标准：

- 所有必验视口和关键状态通过。
- 使用 `@axe-core/playwright` 或阶段 1 明确批准的等价工具，在指定 Playwright 项目中扫描关键页面；无 serious/critical 问题，人工键盘与语义检查通过。
- 儿童观察记录完成，阻断性问题已修复或明确停止发布。

### 阶段 9：隔离 E2E、预览部署、最终验收与生产发布

目标：在不污染生产数据的条件下完成真实 Worker + D1 闭环和安全验收。

任务：

1. 扩展阶段 1 已建立的 `test:e2e:ui`，用确定性 page route fixture 覆盖全部视觉与异常状态。
2. 扩展并运行阶段 1 已建立的 `test:e2e:stack`，覆盖完整 admin → learner 闭环；继续使用独占端口、全新临时 D1、migration 和自动销毁。
3. 让 `test:e2e` 聚合 UI 与本地真实栈两轨；启动后继续验证 health、预期环境标识和测试数据库身份，禁止静默连接旧开发服务或生产 D1。
4. Access service token 只允许在关闭 trace/HAR 的认证 setup request context 中使用，并优先交换为短时测试会话；临时 storage state 只放 suite 临时目录并在结束后销毁。含认证的浏览器 project 默认不保留 trace/HAR；如为调试临时开启，必须先脱敏并扫描通过，未经脱敏的失败产物不得上传或归档。
5. 复用并复核阶段 8 前已另行授权建立的显式 `--env preview` 或独立配置文件驱动的 preview Worker/D1；Worker 名称、URL、环境标识和 D1 身份均不得命中生产。远端预览 smoke 使用 `pnpm test:preview`，不混进无凭证的普通本地测试。
6. `test:preview` 在执行任何写流程前必须校验 preview URL、Worker 名称、环境标识和 D1 身份；任一值缺失、回退默认配置或命中 `eng-learn-prod` 时立即失败退出。preview 命令和 binding 未落地前，preview 验收保持阻断。
7. 使用预览环境和独立测试数据跑完整 admin → learner 流程，并在结束后清理或销毁测试数据环境。
8. 复核 `/admin` 与 `/api/admin/*` 同时受保护，app 会话只能访问所属 course/session/task。
9. 在不加载 `.dev.vars` 的干净 checkout/隔离环境生成发布构建，检查 client bundle、Worker 上传清单、浏览器 storage、`test-results`、trace/HAR 和其他失败产物，不包含管理员 secret、Access header、Cookie、学习码或未提交答案。
10. 在不加载 `.dev.vars` 的干净 checkout/隔离环境运行 Wrangler 类型检查；禁止用本机 secret 派生结果改写并提交 `worker-configuration.d.ts`。
11. 按固定顺序运行全量门禁。
12. 生产只执行最小 smoke，不批量写测试数据。
13. 记录上一 Worker version、回滚步骤，并输出 `pdoc/report/REPORT_*`；若安全、契约或验收仍冲突，输出 `REPORT_CONFLICT_*` 并停止发布。

最终命令顺序：

```text
pnpm typecheck
pnpm lint
pnpm test
pnpm test:e2e  # 聚合 test:e2e:ui 与 test:e2e:stack
pnpm build
pnpm exec wrangler types --check  # 在不加载 .dev.vars 的隔离环境执行
pnpm cf:check
pnpm exec wrangler deploy --dry-run --env preview
```

开发单个失败场景时可分别运行 `pnpm test:e2e:ui` 或 `pnpm test:e2e:stack`，最终门禁只运行聚合命令一次。preview dry-run 必须显式指定 preview 环境，不能回退默认生产配置；部署 preview 后再以受控凭证运行 `pnpm test:preview`。只有 preview 全部通过后，才另行对默认生产配置执行 `pnpm exec wrangler deploy --dry-run` 并进入生产发布审批。`pnpm build` 必须先于 `pnpm cf:check`，避免 `dist/client` 尚未生成造成伪失败。

验收标准：

- 全量门禁通过。
- 预览环境真实闭环通过。
- 管理端和学习端安全边界通过。
- 生产 smoke 只验证入口、认证和一条无破坏读取路径。
- 回滚目标和操作步骤可执行。

## 12. 端到端验收场景

### 12.1 管理端正常闭环

Given：管理员已通过受保护入口进入，准备 20 个字段完整且不重复的测试词。

When：导入、构建、编辑/批准所需项目、发布并创建课程。

Then：

1. 服务端创建一个草稿版本、20 个词和 4 个分组。
2. 页面显示真实覆盖率，不显示审阅稿示例数字。
3. 覆盖率达标后才能发布。
4. 发布后页面只读，刷新后仍只读。
5. course 绑定该 published source version。
6. 页面展示真实学习码且不写入日志。

### 12.2 管理端发布阻断

Given：测试词中至少一个词缺少 exampleSentence。

When：构建练习项目。

Then：

1. S3、S4、S5 缺口按当前真实规则显示。
2. 发布按钮禁用并列出词、阶段和原因。
3. 绕过 UI 直接调用 publish API 仍被拒绝。
4. 通过阶段 0 已确认的草稿修正或下一版本流程处理缺口后，重新读取覆盖率，阻断真实消失。

### 12.3 学习端课时闭环

Given：一个 learner 和一个绑定 published version 的 course。

When：输入学习码、开始 Lesson 1、完成任务并进入报告。

Then：

1. 页面显示服务端 `currentLessonNo = 1`。
2. 任务由 lesson task 快照渲染，响应不提前包含正确答案。
3. 完成后服务端推进到 Lesson 2，页面重新读取最新状态。
4. 服务端注入时间测试证明停学五天后仍是 Lesson 2；浏览器端改变日期不改变页面所显示的服务端课时。

### 12.4 失败、重复与恢复

Given：当前课进行中。

When：断网提交、恢复重试、双击提交、刷新、双击完成。

Then：

1. 断网时保留答案和当前 task，不推进。
2. 重试成功后只产生一次有效答题结果。
3. 刷新后恢复同一 session 和完整 task 快照。
4. 双击完成只推进一次课时。
5. 新增回炉任务不会因前端缓存而丢失。
6. 前四题正确、最后一题错误时，新增 required 回流按规则出现；即使完成率为 5/6，required 未完成前仍不能结束；明确 defer 时不伪造 pending。
7. 其他答错位置的首次同词再现符合 3 至 6 道实际完成题的服务端规则，每词总 task 不超过 3，前端不重排也不计算 defer。

### 12.5 权限隔离

Given：一个管理员会话和两个不同 learner 的课程。

When：无认证访问 admin，或 learner A 请求 learner B 的 course/session/task。

Then：

1. 无认证 admin 不加载业务壳和业务数据。
2. learner A 的跨资源请求由服务端拒绝。
3. app 请求不携带管理员凭证。
4. 前端 bundle、storage 和日志中不存在 `ADMIN_API_TOKEN`。

### 12.6 版本与审核生命周期

Given：一个已发布 source version，以及阶段 0 已确认的版本/审核策略。

When：尝试修改已发布内容，并创建下一草稿版本、编辑/批准/禁用草稿项目。

Then：

1. 已发布版本的 UI 无写入口，直接 API 写入也失败。
2. 同 source 新 draft 的 versionNo 正确递增，旧 course 继续绑定旧版本。
3. 只有 approved 项目计入发布覆盖，disabled 和 draft 项目继续阻断发布。
4. 旧 lesson task 的 ID、prompt、顺序和快照不因新版本变化。

## 13. 总体验收标准

### 13.1 功能

- 管理端真实完成导入、构建、练习编辑/批准/禁用、覆盖审阅、发布和课程创建。
- 学习端真实完成学习码、课程首页、S0-S5、反馈、恢复、回炉和报告。
- 页面刷新、重复提交、重复完成和网络失败有确定行为。
- 前端不计算调度、阶段升级、评分含义或发布资格。

### 13.2 视觉

- 两端共享品牌语言，但布局和信息密度明显不同。
- 学习端一屏一个主要任务，管理端以表格、矩阵和流水线为主。
- 375px 无溢出，320px 长文案可读，1280px 管理核心表格无页面级横向滚动。
- 学习端答案选项达到 56px，学习输入框达到 52px；高对比模式和灰度下状态仍可辨认。
- 不出现竞品品牌资产、自然日打卡、红心、货币或排行榜语义。

### 13.3 无障碍

- 普通文本对比度至少 4.5:1，大文本与非文本控件至少 3:1。
- 状态不只依赖颜色。
- 关键流程可用键盘完成，焦点始终可见且顺序合理。
- 图标按钮具有可访问名称和 tooltip，关键操作不依赖 hover。
- S4 有非拖拽操作路径。
- reduced-motion 下无信息丢失。
- 学习端高频目标达到 48px，管理端达到 40px。

### 13.4 安全与数据一致性

- `/admin` 和 `/api/admin/*` 同时受保护。
- 浏览器不持有静态管理员 secret；测试 trace/HAR/storage state 和失败产物也不保留 Access token、Cookie 或学习码。
- app 服务端强制 learner/course/session/task 归属。
- published source version 不可原地修改。
- 已开始课程继续绑定旧版本，已生成 lesson task 快照不变化。
- 失败和重复请求不造成重复日志或错误推进。

### 13.5 工程门禁

- Vue SFC 纳入 lint。
- 纯逻辑、组件、API 契约和 E2E 均有真实测试。
- `typecheck → lint → test → e2e → build → Cloudflare check/dry-run` 全部通过。
- mocked UI E2E、本地隔离真实栈 E2E 和受控 preview smoke 职责分离且均有可执行命令。
- 没有未解释的新运行时依赖、死路由、硬编码审阅数据或绕过 schema 的断言。
- 最终 `REPORT_*` 已输出；仍有冲突时使用 `REPORT_CONFLICT_*` 并停止发布。

## 14. 风险、控制与停止条件

| 风险 | 等级 | 控制 | 停止条件 |
| --- | --- | --- | --- |
| 管理员 secret 进入浏览器或测试产物 | 高 | Access/HttpOnly 会话；认证 setup 关闭 trace/HAR；临时 storage state；扫描 bundle/storage/test artifacts | 发现任何静态 secret、Access header 或会话 Cookie 立即停止发布、销毁产物并轮换 |
| learner 可访问他人课程 | 高 | 服务端 session 与资源归属校验 | 跨资源请求未被拒绝，停止 app 发布 |
| 前端猜 task prompt | 高 | 判别联合 schema、运行时 parse、穷尽 renderer | 任一 renderer 依赖 `as` 猜结构，停止该切片 |
| mock 与真实 API 漂移 | 高 | 共享 fixture + 真实 Worker/D1 E2E | 真实闭环未过，不得以 mocked E2E 宣称完成 |
| 页面刷新丢业务状态 | 高 | 服务端读取/恢复契约，Pinia 非权威 | 只能靠 localStorage 恢复时停止集成 |
| 错词间隔、次数上限或完成规则失真 | 高 | 服务端 v2 队列策略、3-6 实际间隔、每词总 task 上限 3、answer 后重取权威 session/task、最后一题答错与五词全错反例 | 回流不可见、间隔/上限错误、required 未完可结课或 defer 后仍无限拉长则停止 |
| 版本/审核生命周期未实现 | 高 | 按上位计划实现同 source 新版本和 draft/approved/disabled 状态流 | 未通过 service/API 测试时停止 walking skeleton publish 与完整管理端 |
| 发布版本被前端误导为可编辑 | 高 | 读取服务端状态，UI 只读，API 二次拒绝 | 任何已发布写入成功时停止发布 |
| E2E 误连生产 D1 | 高 | 独占端口、全新本地 D1、环境/数据库身份检查、独立 preview binding | 无法证明数据库身份时立即停止测试 |
| 一次性铺全 UI 导致返工 | 中 | 纵向切片、每片全量回归 | 契约未定仍扩大页面时暂停扩展 |
| CSS 局部值漂移 | 中 | token、基础组件、截图回归 | 同语义出现第二套未解释 token 时收口 |
| 截图测试频繁误报 | 中 | 固定数据/时间/字体/动效，确认后建基线 | 页面未定稿时不建立 baseline |
| 移动端软键盘遮挡 | 中 | 真实设备、动态视口、安全区检查 | 核心输入或主操作被遮挡时停止发布 |
| 儿童不理解交互 | 中 | 至少 3 名目标年龄参与者、匿名观察和复测 | 任一严重阻断未修复，或无法安排真实参与者却宣称儿童可用性通过时停止发布 |
| 范围扩张 | 高 | 非目标清单和新功能准入 | 出现 AI/R2/支付/排行榜等需求时另立计划 |

## 15. 回滚策略

### 15.1 前端与 Worker

- 每个纵向切片独立小提交，行为测试与实现同提交。
- 不长期维护新旧两套 UI，也不为本轮建立通用 feature flag 系统。
- 预览或生产异常时回滚到上一已验证 Worker version。
- token、组件和对应页面一起回滚，避免保留半套设计系统。

### 15.2 API 与数据

- 前端依赖的 API 补齐保持 additive，不做破坏性 D1 migration。
- 内容问题不修改已发布版本；在同一 source 下创建下一 draft version 修复。
- 历史 course 继续绑定原 published version。
- 历史 lesson task 快照保持不变，不由前端重新生成。

### 15.3 安全事件

- 若发现管理员令牌进入 bundle、日志或 storage，立即停止部署、回滚并轮换 secret。
- 若发现 learner 越权，立即停止 app 公网入口，修复服务端归属校验后重跑安全验收。
