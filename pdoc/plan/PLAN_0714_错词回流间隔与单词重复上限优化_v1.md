## 2. 计划结论

本次不应只把代码中的 `5-8` 改成 `3-6`。该修改只能解除“五词全错时没有第六个安全词”的原始死锁，仍不能阻止以下问题：

1. 同一个词被多个回流任务反复克隆成 `bridge`，用户实际看到的次数仍可能超过上限。
2. `primary`、`bridge`、`reflux` 的错误处理不一致，部分用户可见错题可能被静默忽略。
3. 少于 4 个安全词的课时仍无法严格提供 3 道间隔题。
4. 达到上限后如果仍生成必答回流，课时仍可能无限增长、永远不能结束。
5. 旧 session 若中途从旧策略切换到新策略，刷新、恢复和 Worker 回滚会产生双重语义。

推荐采用以下完整但聚焦的 v2 规则：

```text
错答后首次同词再现的完整间隔：3 至 6 道题
同一单词同一课最多追加出现：2 次
同一单词同一课最多实际出现：3 次（primary + bridge + reflux）
达到上限或当前队列无合法排程：不再追加，顺延到下一课
完成条件：80% primary + 所有已经生成的 required task 完成
```

该默认值不是数学唯一解。2026-07-14，Solazhu 已确认第 11 节全部参数并明确授权按本计划执行；本地编码和内部验收结果记录在 `REPORT_0714`。

## 3. 目标与非目标

### 3.1 目标

1. 五词连续全错时不存在无题可排或无限回流。
2. 每次实际安排的同词回流都严格满足 3 至 6 道完整间隔。
3. 每个单词在同一课的用户可见任务数有硬上限。
4. 达到上限仍答错时，系统记录结果、安排下一课复习并允许本课有限结束。
5. 刷新、重复提交、并发提交、Worker 重启和 D1 恢复不重复消耗上限。
6. 旧策略 session 与新策略 session 不静默混用。

### 3.2 非目标

- 不修改每组 5 个新词的内容分组规则。
- 不借用尚未激活的未来词。
- 不新增中性 filler 题型、AI 出题或临时伪任务。
- 不重写 S0-S5、StageEngine 的升降级公式或跨课间隔模型。
- 不原地修改已发布 source version、已完成 task 或历史 review log。
- 本次执行授权只覆盖当前 checkout 的本地编码与验证，不授权远端 migration、部署或生产数据操作。

## 4. 冻结的数学与业务定义

### 4.1 间隔公式

位置按学习者实际完成且产生 review log 的可评分任务序列计算。skipped task 不属于学习者实际经历的间隔题。设答错任务位置为 `wrongPosition`，答错后第一次再次完成相同 `wordId` 的位置为 `nextSameWordPosition`：

```text
interveningCount = nextSameWordPosition - wrongPosition - 1
3 <= interveningCount <= 6
```

验证必须扫描答错任务之后所有已完成的 `primary / bridge / reflux`，找到第一个相同 `wordId`；只验证 `role = reflux` 的持久化 `orderIndex` 不算通过。排程阶段可使用 pending 位置模拟未来序列，但所有充当间隔题的 bridge 必须为 required，最终验收必须证明中间每题都实际完成。

五词全错时，合法序列示例为：

```text
A B C D E | A B C D E | A B C D E
```

同词位置差为 5，中间完整隔 4 题。当本课冻结词集合恰好为这 5 个词时，全错必须恰好生成 15 题且不得出现 `deferred_capacity`。

当前实现可能将全部 due word 与下一组 5 个新词放入同一课，因此不能把“15 题”写成所有课的总上限。v2 session 的 `lessonWordSet` 定义为创建 session 时已持久化 primary task 的不同 `wordId` 集合；该集合不可变，本课后续 task 都只能来自它。设 `N = lessonWordSet.size`，则通用上限为：

```text
lessonTaskCount <= 3 * N
```

本计划不额外修改每课 primary 总词数；N 大于 5 的压力课必须按 `3N` 验证。

### 4.2 上限口径

权威口径从当前 session 已持久化的 `lesson_tasks` 派生：

```text
scheduledCount(sessionId, wordId)
  = 该 session 中该 wordId 的全部 task 行数
  = primary + bridge + reflux

MAX_SCHEDULED_TASKS_PER_WORD_PER_LESSON = 3
MAX_EXTRA_APPEARANCES_PER_WORD_PER_LESSON
  = MAX_SCHEDULED_TASKS_PER_WORD_PER_LESSON - 1
```

- pending task 已经占用预算，不能等到答完后才计数。
- skipped task 仍保留不可变快照并占用预算，不能通过跳过重新获得次数；但 skipped 不计入实际间隔题数。
- 不新增可漂移的内存计数或 `user_word_states` 计数字段。
- `review_logs` 用于记录答案，不作为调度上限的唯一事实源。

### 4.3 达到上限或无法合法排程

每次非通过答案只能产生一个确定结果：

1. `scheduled`：在剩余预算内生成一个 required 回流任务；或
2. `deferred_cap`：该词已达到本课出现上限；或
3. `deferred_capacity`：当前安全词与 pending obligation 无法组成合法 3-6 排程。

disposition 必须通过 additive migration 持久化在每个 task 的唯一 answer outcome 上，建议由 review log 承载 `scheduled / deferred_cap / deferred_capacity`。它与 answer、word state 和 task mutation 同批提交，重复提交必须返回首次写入的 disposition。

后两种结果必须满足：

- 当前 answer 和唯一 review log 正常提交。
- 不生成新的 required task。
- 该词最迟在下一课到期，并进入“需继续练习”报告。
- 不因“本课最后一次仍答错”永久阻断结课。
- 该 `wordId` 在本课立即 closed，后续任何 role 都不得再选择或生成它。
- 同词同一时刻最多存在一个 open obligation 或 pending recurrence。

`deferred_capacity` 当且仅当随 Worker 发布的精确 `isQueueFeasible` oracle 证明不存在合法排程。`buildSchedule` 与 oracle 必须为独立实现；测试再用穷举基准交叉验证二者。若 oracle 证明存在解而生产调度器失败，必须返回稳定 `queue_invariant_violation`、零业务写入并使测试失败，不得降级成 defer。

### 4.4 task role 与 StageEngine

- `primary`：继续作为本课唯一完整推进 StageEngine 的任务。
- `reflux`：由上一条同词错答产生，参与 3-6 间隔与每词上限；每个 open obligation 只能由带对应 `refluxSourceTaskId` 的 reflux task 关闭。
- `bridge`：只能来自冻结 `lessonWordSet`，且该词的本 session primary 已完成、没有 pending 同词 task、没有 open obligation、没有被 defer、未 suspended、仍有次数预算。
- bridge 不能命中或关闭既有 obligation；bridge 自身答错可以成为新的 source，但仍受同一总上限。
- 任一用户可见、可评分 role 答错，都进入同一排程判断；不得因其内部 role 不同而静默忽略。
- `bridge / reflux` 不重复升级或降级 stage，避免同一课多次计权。
- 非 primary 最终答错时，由 StageEngine 的专用 `deferToNextLesson` transition 将 `nextDueLessonNo` 收紧到 `currentLessonNo + 1`；CourseRuntime 不直接计算该字段，不提高 mastery，不重复升级。

### 4.5 完成规则

保留现有最小改动语义：

```text
完成至少 80% primary
+ 所有已经生成的 required task 已完成
= 允许结课
```

`deferred_cap` 和 `deferred_capacity` 不是 pending obligation；它们进入下一课，不继续拉长本课。

## 5. 调度实现方案

### 5.1 从局部随机插入改为全局可行性排程

现有实现按单个错题随机选择 gap，再补 bridge。新实现改为一个服务端纯函数；权威输入为：

```text
LessonQueueSnapshot
  = session queue policy
  + frozen lessonWordSet
  + ordered lesson tasks
  + submitted scores and persisted dispositions
```

调度器只重算 pending tail。每条 completed 记录在写入时验证，后续不得只靠 task role 猜测历史答案。统一处理规则为：

1. completed/skipped 前缀不可重排。
2. 只允许原子调整同 session 的 pending task 顺序和新增 task。
3. 每个 obligation 计算最早位置 `source + 4`、最晚位置 `source + 7`。
4. 优先安排最早截止的 obligation，不再先随机选 gap；相同条件依次按 deadline、source answer 顺序、冻结 word 顺序、task ID 决定，保证恢复结果确定。
5. bridge 只能使用第 4.4 节的合法候选；生成前还要模拟该 bridge 自身答错时是否仍可 scheduled 或明确 defer。
6. bridge 不得与任何 open obligation 同词，也不得排在已有 pending 同词 task 之前。
7. 新增 obligation 前先调用生产 `isQueueFeasible` 精确 oracle；oracle 无解才允许 `deferred_capacity`，随后由独立的 `buildSchedule` 构造队列。
8. oracle 有解但生产调度失败、队列 source 丢失、同词存在两个 open obligation或 cap 已被突破时，返回 `queue_invariant_violation` 并零写入。
9. 每次排程完成后按 snapshot 中的 score/disposition 扫描相关错答 source 的首次同词再现，作为服务层 invariant。

五词全错应自然排成四道间隔的循环；不要求为了“随机性”使用 3、5、6。

### 5.2 安全词不足

3 道最小间隔意味着要对任意答题结果保证严格同课回流，冻结集合至少需要 4 个可安全轮换的不同 `wordId`。当前一词短词库、尾组或 due-only 课可能只有 1 至 3 个 primary。

处理顺序：

1. session 创建时从已持久化 primary task 的不同 `wordId` 冻结 `lessonWordSet`；后续可直接从这些 primary 行重建，不得在答题途中动态引入新 wordId。
2. bridge 只能克隆集合内已有不可变 task 快照，不读取未来词或最新发布内容重新造题。
3. `N >= 4` 时必须先用 oracle 证明具体状态无解，不能只因生产算法排不出就 defer。
4. `N < 4` 时进入明确的 short-pool 降级：当前 primary 可正常完成，错词使用 `deferred_capacity`，本课不再出现；不伪造题、不借未来词、不让单个课时死锁。
5. 上限按课时重置，不是单词终身上限。short-pool 词下一课仍按 StageEngine 到期，但本计划只保证“下一课到期”，不承诺下一课能够形成同课 3-6 回流；若产品要求该保证，必须另立“补足安全词池或中性 filler”方案。

连续两课 short-pool 测试必须证明每课都有限结束、状态可解释，不能把“下一课到期”误写成“下一课必定完成回流”。

### 5.3 幂等、并发与事务

- cap 与可行性判断基于同一份 `LessonQueueSnapshot`；repository 以 current pending task、唯一 review log 和唯一 source child 作为 winner 守卫，answer、disposition、word-state transition 与 task mutation 只由 winner 同批提交。
- 相同 task 的重复提交返回已有结果，不再次计算或消耗 cap。
- 并发提交只允许一个 winner 创建 obligation；loser 读取 winner 的结果。
- D1 唯一约束继续保证一个 source 最多一个 reflux child。
- `queue_invariant_violation` 和 repository 冲突路径必须在 answer、review log、word state 和 lesson task 四类写入上保持零部分提交。

### 5.4 session 策略版本

生产级实现新增 session 级队列策略版本和三态写入开关，例如：

```text
v1_5_8_unbounded
v2_3_6_cap3

queueWriteMode = legacy_v1 | v2 | disabled
```

采用 expand-first 两阶段发布：

1. 先部署 additive migration 和能双读 v1/v2 的兼容 Worker，`queueWriteMode = legacy_v1`，仍只创建 v1。
2. 只读审计 started v1 session。结果不为 0 时继续阻断 v2，由 Solazhu 另行授权并验证完成、放弃重开或兼容迁移；当前仓库没有现成的放弃重开能力，不能把它写成已可执行操作。
3. started v1 清零或处置完成后，再切换 `queueWriteMode = v2`，新 session 写 v2。
4. 已启动 v1 不静默切换到 v2，也不重建其题目快照。
5. 产生 v2 session 后，唯一回退目标是双读兼容 Worker，并切换 `queueWriteMode = disabled`：禁止创建新 session，但允许按各自 policy 恢复已有 started v1/v2 session；不得回滚到只理解 5-8 的当前旧 Worker，也不得恢复创建已废弃的 v1 session。

| oracle | `isQueueFeasible`、`buildSchedule`、测试穷举基准 | 三方结论一致；oracle 有解必须 scheduled |
| 完成 | 80% primary、required 未完成、cap exhausted | required 未完成拒绝；cap exhausted 不永久阻断 |
| 恢复 | 回流中刷新、Worker 重启 | task 顺序、预算、当前题不变 |
| 幂等 | 双击、相同 sessionId/taskId 重试、响应丢失 | 一条 log、一个 disposition、一个 child、一次 cap 消耗 |
| 并发 | 两个 answer 同时提交 | 一个 winner；loser 返回权威结果 |
| D1 | answer + reorder + defer | 原子提交；错误路径零部分写入 |
| migration | 旧 session 回填、重复应用、v2 新 session | 旧行为标 v1；新行为标 v2；additive 且可重复验证 |
| 兼容 | v1/v2 started 与三态 queueWriteMode | 不混用策略；disabled 禁止新 session 但可恢复已有 session |
| 报告 | 达 cap 或容量不足仍错 | 稳定进入需继续练习，下一课可见 |
| 浏览器 | 五词全错、刷新、完成 | 有限完成且 UI/服务端一致 |

可增加基于属性的序列测试，在受控任务规模内穷举或随机生成对错序列，统一断言：

```text
每词 task 数 <= 3
每个 scheduled 回流之间实际完成的题数在 3-6
completed/skipped task 不重排
skipped task 不计入 interveningCount
队列长度有限
合法输入最终得到 next task、defer 或 complete
腐坏状态只能得到 queue_invariant_violation 且零写入
oracle 有解时不得 deferred_capacity
```

## 8. 验收标准

只有同时满足以下条件，原五词回流阻断才可关闭：

1. 上位计划、代码规范、前端计划和执行报告不存在互相冲突的现行口径。
2. 五词全错按 4 道实际间隔运行，恰好 15 题且没有 capacity defer。
3. 所有已安排回流的首次同词再现之间有 3-6 道已完成题；skipped 不冒充间隔。
4. 每个单词同课所有用户可见 task 总数不超过 3，任意 N 词课总 task 不超过 `3N`。
5. 第三次仍错不会生成第四次，能进入下一课并允许本课结束。
6. 1 至 3 个冻结词进入明确 short-pool 降级，不产生伪题、未来词或单课死锁，也不虚假承诺下一课能同课回流。
7. pending required 未完成仍不能结课；defer 不伪装为 pending obligation。
8. disposition 已持久化；defer 后同词本课 closed，重复提交、并发、刷新和重启不会突破 cap 或重复写日志。
9. 内存、SQLite/D1、Worker API 和浏览器使用相同语义。
10. published content、历史 task 快照和完成记录未被原地改写。
11. 生产 oracle、队列构造器与测试穷举基准结论一致；oracle 有解时不得 capacity defer，构造失败与腐坏队列稳定 fail-closed、零写入。
12. 新增 migration 为 additive；started v1 未清零或未获处置授权时继续阻断 v2；双读兼容与开关回退通过验证。
13. 全量自动化门禁通过，并在隔离 preview 复核后才允许生产审批。

## 9. 风险与回滚

| 风险 | 控制方式 |
| --- | --- |
| 只改常量，bridge 仍提前撞词 | 对首次相同 `wordId` 做全 role invariant |
| task cap 与 review log 数不一致 | cap 只从持久化 lesson task 快照派生 |
| bridge 消耗预算后挤掉真正回流 | 新增前模拟全部 pending obligation，可行才提交 |
| 算法失败被伪装成容量不足 | 生产精确 oracle、独立构造器和测试穷举基准三方校验 |
| defer 后同词再次被 bridge 选中 | disposition 持久化并将同词本课 closed |
| 短词库无安全题 | short-pool 明确降级；只承诺下课到期，不承诺下课同课回流 |
| 非 primary 错误被遗忘 | 写 review log，由 StageEngine transition 收紧下一课到期 |
| 旧新策略中途混用 | session policy version + started-session 只读审计 |
| Worker 回滚后不理解 v2 | 只回滚到同时读取 v1/v2 的兼容版本 |
| 队列算法修改影响完成规则 | repository 与 service 双层完成门禁回归 |

回退先切换 `queueWriteMode = disabled`，再切回双读兼容 Worker；禁止创建新 session，但允许恢复已有 started session。不删除 v2 task、不 down-migrate D1、不改写已完成 session。当前只理解 v1 的旧 Worker不是合法回滚目标。
