## 2. 当前事实与根因

### 2.1 已确认的生产现象

2026-07-17 对生产页面和远端 D1 的只读核验得到以下事实：

1. 用户选择 CSV 后，浏览器只在本地解析和预览，不会写服务端。
2. 用户点击“创建草稿版本”后，页面才第一次调用 `POST /api/admin/source-versions/import`。
3. 本次 118 词导入连续两次得到 HTTP 500，响应都是合法 API 错误：

```json
{
  "ok": false,
  "error": {
    "code": "internal_error",
    "message": "Unexpected server error"
  }
}
```

4. 这不是浏览器断网，也不是请求结果丢失。
5. 生产 D1 的 `d1_migrations` 只记录到 `0010`，本地已随新代码存在的 `0011_add_progressive_context_model.sql` 未应用。
6. 当前 Worker 已读取 `source_versions.content_model`、`words.example_phrase` 和 `words.example_sentence_extended`，生产 D1 中却没有这些列。
7. 远端数据库核验时：`word_sources=0`、`source_versions=0`、`words=0`、`admin_operations=0`。两次失败均已回滚，没有草稿、半成品或重复数据。
8. 118 词低于现有 500 行和 256 KiB 导入上限；本次失败不是容量问题。

### 2.2 故障链

```text
新 Worker 依赖 0011 新字段
  -> 自动部署已将新 Worker 发布到生产
  -> 生产 D1 仍停在 0010
  -> 导入路径第一次访问新字段时 SQL 失败
  -> API 返回明确的 HTTP 500 / internal_error
  -> 前端把 internal_error 误归类为“结果未知”
  -> 用户被要求手动重试同一个必然失败的请求
```

根因有两层：

- 直接根因：生产 D1 漏应用 `0011`。
- 系统根因：发布流程没有“数据库迁移齐全后才能发布 Worker”的硬门禁；前端又把明确的服务端失败和真正的传输不确定混在一个“结果未知”状态中。

### 2.3 草稿不是故障

导入后创建草稿必须保留。项目冻结的内容链路是：

```text
导入词表
  -> 创建可编辑草稿版本
  -> 构建练习
  -> 人工审核
  -> 发布不可变版本
  -> 课程只绑定已发布版本
```

如果导入后直接发布，将绕过练习构建、覆盖率检查和人工审核，并破坏“已发布版本不可变”的硬边界。

本次要修的是文案和操作语义：按钮改为“导入并创建草稿”，让用户清楚一次点击同时完成导入和草稿创建；不是取消草稿。

## 3. 修复目标与完成标准

### 3.1 总目标

恢复生产导入能力，并把词表导入收口成“一次用户确认、系统自动得出权威结果”的流程，同时建立数据库迁移发布门禁，防止同类故障再次进入生产。

### 3.2 完成必须同时满足

1. 生产 D1 已应用 `0011`，新字段和不可变触发器存在。
2. 用户选择 CSV 后仍先做本地预览；只有一次明确的服务端写入确认。
3. 主按钮文案为“导入并创建草稿”。
4. 正常导入不再要求用户额外点击“安全重试”。
5. 明确的校验、业务、权限、数据库版本和服务端失败显示为明确失败，不显示“结果未知”。
6. 真正可能发生“服务端已提交但响应丢失”的情况由系统用同一个 operation token 自动确认，不要求用户理解幂等机制。
7. 新词库和“同词库下一版本”两种模式都使用相同的 operation token、请求指纹、原子账本和自动确认语义。
8. 同一 operation token + 同一请求无论重放多少次，最多创建一个版本；同一 token + 不同请求返回 `idempotency_conflict`。
9. 请求指纹覆盖六列全部内容以及导入模式、词库名称或 source ID，不能漏掉 `examplePhrase` 和 `exampleSentenceExtended`。
10. 页面刷新后，同一浏览器标签页能恢复尚未完成的确认，不生成新 token。
11. 生产发布前若远端 D1 存在未应用 migration，Worker 部署必须失败关闭。
12. 118 词真实导入后，服务端权威结果为 1 个 source、1 个 draft version、118 个 words、24 个 groups、1 条导入 operation；不存在重复或半完成数据。

## 5. 冻结的结果语义

### 5.1 用户可见状态

| 状态 | 含义 | 用户动作 | 系统动作 |
| --- | --- | --- | --- |
| 预览中 | 浏览器本地解析 CSV | 等待 | 不发网络请求 |
| 预览通过 | CSV 可提交 | 点击一次“导入并创建草稿” | 生成一次 operation token 并提交 |
| 正在导入 | 服务端正在处理首次请求 | 无 | 禁止重复点击 |
| 正在确认 | 响应可能丢失或服务暂时不可达 | 无 | 自动用同一 token 和同一 payload 重放 |
| 导入成功 | 服务端返回权威版本、词数和组数 | 可进入草稿详情 | 清除 pending command，刷新列表 |
| 导入失败 | 服务端明确拒绝或确认未提交 | 按错误修正后重新发起 | 清除 pending command，保留 CSV 预览 |

“正在确认”是短暂的进行态，不是导入结果。界面不再出现“词表导入结果未知”标题。

### 5.2 API 错误分类

| 类别 | 示例 | 是否可能已写入 | 前端处理 |
| --- | --- | --- | --- |
| 明确拒绝 | `validation_error`、`source_draft_exists`、`not_found`、`idempotency_conflict` | 否 | 立即显示明确失败 |
| 数据库版本未就绪 | 新增 `schema_not_ready` | 否 | 明确提示服务端未升级；不盲目重试 |
| 已确认成功 | 合法 2xx success envelope | 是，且结果完整 | 显示版本号、词数、组数 |
| 需要自动确认 | 新增 `import_reconcile_required`，或浏览器未收到合法响应 | 尚不能从当前响应判断 | 进入“正在确认”，自动重放同一命令 |
| 普通内部失败 | `internal_error`，且服务端已完成账本核验并确认没有提交 | 否 | 显示服务端失败，不归类为未知 |

不得再仅凭 HTTP 500、503 或异常类名猜测业务结果。服务端必须先把“明确未提交”和“需要幂等确认”转换成稳定错误码，前端只按稳定契约处理。

### 5.3 自动确认策略

1. 第一次提交前，把完整、已通过 schema 的导入命令保存到当前标签页的 `sessionStorage`；不写 `localStorage`，不作为服务端权威数据。
2. 首次响应属于 `import_reconcile_required`、网络失败或响应契约损坏时，页面立即自动重放同一命令。
3. 自动确认使用同一个 token 和逐字节等价的业务 payload，不生成新 token，不重读当前表单拼新请求。
4. 连续暂时不可用时采用逐步增大但有最大间隔的退避；页面可见时继续自动确认，页面隐藏时暂停，并在重新可见、浏览器 `online` 事件或同标签页刷新后自动继续。
5. 得到权威成功或明确失败后立即清除 pending command。
6. session 数据损坏、过大或不符合当前 schema 时失败关闭并清除，不自动提交无法证明来源的内容。
7. 关闭标签页结束恢复能力；不扩展为离线同步系统。

### 5.4 两种导入模式统一幂等

新词库与下一版本统一要求：

```text
operation token
  + import mode
  + sourceName 或 sourceId
  + 六列完整 words payload
  -> 版本化请求指纹
  -> admin operation ledger
  -> source/version/words/groups 同批原子写入
```

实现保持现有 `admin_operations` 表的全局 token 唯一性，不新增平行幂等表。持久化兼容现有 `create_source` kind，由 `target_id='new-source'` 或真实 source ID 区分两种导入目标；代码类型和注释改为“词库版本导入 operation”，避免继续把它误解为只适用于新词库。

请求指纹升级时必须兼容既有 v1 记录：

- 新请求使用覆盖六列和 mode/target 的 v2 指纹。
- 查询到既有 v1 记录时按旧算法校验并允许精确重放。
- 不允许升级后把旧 token 误判成新请求，或把只改变新语境字段的 payload 当成同一请求。

### 阶段 2：导入契约和服务端幂等

目标：让服务端成为导入结果唯一事实来源，并让两种模式都能安全自动重放。

红测：

1. `next_version` 没有 operation token 时 schema 拒绝。
2. 新词库和下一版本使用同一 token + 同一 payload 重放，只返回第一次结果，各自只创建一个版本。
3. 同一 token 改 mode、sourceName、sourceId、任一单词字段或顺序时返回 `idempotency_conflict`。
4. 只改变 `examplePhrase` 或 `exampleSentenceExtended` 也必须发生指纹冲突。
5. 注入 D1 batch 中途失败时，operation、source/version、words、groups 全部为零写入；随后同 token 可成功重试。
6. 模拟业务数据已提交但首次响应读取失败时，服务端通过账本重放返回第一次结果。
7. 账本暂时不可读时返回 `import_reconcile_required`，不伪装成明确失败。
8. 远端 schema 低于最低 migration 时返回 `schema_not_ready`，业务表零写入。
9. 既有 v1 指纹记录仍可精确重放，不被 v2 算法破坏。

最小实现：

1. `importSourceVersionCommandSchema` 的两个分支都要求 operation token。
2. 把 `importNextVersion` 改为与新词库相同的 idempotent service 接口。
3. 统一导入 request fingerprint，覆盖 mode、target 和六列完整 payload。
4. 复用现有 operation ledger 和 D1 batch，把导入结果账本与业务记录原子提交。
5. 增加 v1/v2 指纹兼容读取，不回写或破坏既有账本。
6. 增加 schema readiness 检查，将 migration 漏应用转换为稳定 `schema_not_ready`。
7. 只把无法完成账本核验的情况转换为 `import_reconcile_required`。
8. 保持 route 只解析契约和调用 service；D1 查询继续位于 repository/基础设施层。

阶段验收：

- 内存、D1 adapter、Worker API 三层行为一致。
- 新词库和下一版本不再有不同的未知恢复策略。
- 所有失败注入都能证明零写入或精确一次提交。

### 阶段 3：前端一次确认与自动结果收敛

目标：用户只做一次写入确认，普通流程不再理解或操作幂等重试。

红测：

1. 选择文件只产生本地预览，不发导入请求。
2. 主按钮显示“导入并创建草稿”。
3. 一次点击后正常成功，显示服务端版本号、词数和组数。
4. `schema_not_ready`、validation、draft conflict 和已核验 internal error 显示明确失败，不出现“结果未知”。
5. 首次响应丢失后页面自动用完全相同的 token 和 payload 重放，无第二次用户点击。
6. 自动重放成功后数据库只有一个版本，页面显示成功。
7. 自动确认期间表单、模式、文件和主按钮不可重复提交。
8. `next_version` 与 `new_source` 走同一状态机。
9. 同标签页刷新后自动恢复 pending command，仍使用原 token。
10. malformed session 数据不会触发网络写入。
11. 终态后 session pending 数据被清除。

最小实现：

1. 将导入状态收口为显式状态机，不再以多个 boolean 组合推导结果。
2. 删除 `isUnknownResult` 对所有 500/503 的宽泛归类。
3. 删除普通流程的“安全重试同一次导入”按钮和“结果未知”错误卡。
4. 增加独立 import recovery 模块，负责保存、校验、重放和清除 pending command；页面不直接散写 storage 逻辑。
5. `executeImport` 在需要确认时自动调用同一 command，而不是重新从当前表单创建 command。
6. 成功后刷新版本列表；列表刷新失败不能把已确认导入改写成失败。
7. 明确失败后保留已验证 CSV 预览，便于修正服务端条件后重新提交。

阶段验收：

- 普通成功路径只有一次用户点击。
- 响应丢失路径可以有多次同 token 网络请求，但只有一次业务写入、零次额外用户点击。
- 页面不再出现“导入结果未知”文案。

### 阶段 4：远端 migration 发布门禁

目标：任何依赖新 schema 的 Worker 都不能先于对应 migration 进入生产流量。

红测：

1. 本地 migration 比远端多一个时，发布预检以非零退出。
2. 本地与远端 migration 名称和顺序完全一致时，预检通过。
3. 数据库名称、ID 或环境不匹配时失败关闭。
4. Wrangler 输出不可解析、认证失败或远端查询失败时失败关闭。
5. 预检脚本只读，不执行 migration 或业务写入。

最小实现：

1. 新增只读 `scripts/check-remote-d1-migrations.mjs`：
   - 从 `wrangler.jsonc` 解析目标 database name 和 ID。
   - 从 `migrations/` 读取本地 migration 清单。
   - 只读查询远端 `d1_migrations`。
   - 比较名称、顺序和完整性，任一不一致即非零退出。
2. 新增明确分离的 package scripts：
   - `db:migrations:check:remote`：只读门禁。
   - `db:migrate:remote`：显式生产写入，只能在单独授权后运行。
   - `release:deploy`：先执行只读 migration 门禁，再执行 `wrangler deploy --strict`。
3. Cloudflare Workers Builds 的生产 deploy command 改为 `pnpm release:deploy`。
4. 不把 `db:migrate:remote` 放进普通 build 或自动 deploy；数据库生产变更继续需要人工批准。
5. 如果 Workers Builds 当前 token 无法以最小权限只读 D1，则禁止回退到无门禁自动部署；改为 `wrangler versions upload` 不自动激活，生产由获批的手动 `release:deploy` 完成。

## 8. 发布、回滚与数据安全

### 8.1 发布顺序

生产顺序固定为：

```text
additive migration
  -> 远端 schema 探针
  -> Worker deploy
  -> health/list smoke
  -> 一次真实导入
  -> D1 终态核验
```

禁止再次执行“先部署依赖新字段的 Worker，再补 migration”。

### 8.2 回滚

- `0011` 是 additive migration，旧 Worker 不依赖新字段也能继续运行；代码回滚时保留 `0011`，不删除列或触发器。
- Worker 回归：回滚到上一稳定 deployment，重新运行 health 和版本列表 smoke。
- migration apply 自身失败：停止发布；按 Cloudflare migration 事务结果确认该 migration 已回滚，不能继续部署 Worker。
- 迁移后发现非预期数据或 schema 破坏：停止所有写入，使用迁移前 bookmark 评估 D1 Time Travel restore。
- Time Travel restore 会覆盖当前数据库，属于破坏性生产操作，必须再次获得 Solazhu 明确授权；不得把它作为普通自动回滚步骤。
- 导入故障：operation ledger 和业务记录同批提交；不通过手工删行修复重复，先用 token、指纹和权威计数定位。

### 8.3 风险与控制

| 风险 | 控制 | 停止条件 |
| --- | --- | --- |
| 自动重放产生重复版本 | 两种模式统一 operation token、完整指纹和原子账本 | 同 token 出现两条版本 |
| v2 指纹破坏既有 operation | 双算法兼容读取和专门回归 | 既有精确重放变成 conflict |
| session 恢复提交陈旧或损坏命令 | sessionStorage、schema 校验、终态立即清除 | malformed 数据触发网络请求 |
| 明确失败仍被误叫未知 | 稳定错误码与状态机测试 | 500/schema error 出现 unknown 卡片 |
| 真正断网时伪报失败 | 使用进行中的自动确认态，不制造终态 | 无服务端证据却显示“未创建” |
| 自动构建 token 权限不足 | 最小只读权限；否则关闭自动激活并手动 gated deploy | 回退到无门禁生产部署 |
| migration 应用到错误数据库 | 使用固定 database name、校验 ID、执行前后列出 migration | name/ID/environment 任一不符 |
| 回滚误覆盖新数据 | bookmark、停止写入、单独授权 Time Travel | 未核对时间点就 restore |

## 12. 后续代码审查修正（2026-07-17）

### 12.1 复核发现

首次验收完成后的独立代码复核发现三个仍需收口的缺口：

1. operation ledger 已成功查空后，`assertImportSchemaReady()` 的普通依赖失败仍被转换为 `import_reconcile_required`。这会把已明确未写入的数据库不可用错误误报为需要自动确认，违反本计划第 5.2 节“只有无法完成账本核验才进入自动确认”的边界。
2. `adminOperationCrypto.ts` 重复声明了 `ImportWordInput` 的六个字段。共享类型未来增加字段时，请求指纹可能继续编译但漏掉新字段。
3. 原整栈 runner 总是先应用全部 migration，无法真实证明“只到 0010、缺 0011”的 Worker + D1 行为；响应丢失用例也只核对版本数量，没有直接核对 `admin_operations` 数量。

这些发现不改变既有生产数据验收结论，也不需要重新执行远端 migration、部署或生产写入。

### 12.2 修正方案

1. readiness 探针在 ledger 已查空后保留原始稳定错误：缺列继续返回 `schema_not_ready`，普通依赖失败继续返回 `dependency_failure`；不再改写为 `import_reconcile_required`。
2. 请求指纹直接复用共享 `ImportWordInput`，并用 `Record<keyof ImportWordInput, string>` 对序列化字段做编译期穷尽约束。
3. 用固定 golden hash 锁定既有 v2 指纹字节兼容性。
4. 整栈 runner 先暂存 `0011`，对真实 0001–0010 D1 运行缺迁移用例；确认 HTTP 503 / `schema_not_ready` 且 source、version、word、group、operation 全部零写入后，再恢复并应用 `0011`，重启同一隔离 D1 运行完整闭环。
5. 本地 E2E Worker 增加受 loopback、`local-e2e` 环境和随机数据库 sentinel 三重约束的只读证据接口；该接口不进入生产 Worker。
6. 响应丢失用例按 source ID 直接核对 2 个版本、40 个词、8 个分组和 2 条导入 operation，证明两个 token 各精确提交一次。

### 12.3 新增完成标准

- ledger 已知为空时，readiness 普通依赖失败必须保持 `dependency_failure`。
- 既有六列 v2 指纹必须保持 `sha256:c72cff2b43def325abb34ec452584cdb3e3fed96ce23dfccfe3507f849052c31`。
- 共享 `ImportWordInput` 增加字段时，指纹序列化必须在编译期要求同步处理。
- 真实 0010 D1 上的导入必须返回 `schema_not_ready`，且五类导入状态计数全部为零。
- 同一 source 的新建导入和下一版本各发生一次响应丢失重放后，必须只有 2 条 import operation 和 2 个版本。
