## 2. 验收结论

新导入词库已经使用三层语境内容模型，六个学习阶段按以下顺序构建：

```text
S0  单词 + 释义 + 最短短语
S1  多项选择识别
S2  主动默写单词
S3  基础句填空
S4  扩展句拼装
S5  独立输出后对照扩展句
```

历史词库继续使用原 S1/S2 题型顺序和单例句模型；既有 lesson task 快照不被重建。错题回流的 3–6 题间隔、每词同课最多 3 次和队列逻辑没有业务修改，并通过原有回归与真实整栈测试。

## 3. 实施范围

### 3.1 数据与版本兼容

- 新增 `v1_single_sentence` 与 `v2_progressive_context` 内容模型。
- 新增 `0011_add_progressive_context_model.sql`：
  - 历史词库默认回填为 v1。
  - words 增加 `example_phrase` 与 `example_sentence_extended`。
  - 数据库约束内容模型枚举值，并阻止版本内容模型被改写。
- 新导入版本明确写入 v2；D1 与内存仓库完整读写三层语境。

### 3.2 CSV 与管理端

- 模板和严格导入表头统一为：

```text
word,meaning,examplePhrase,exampleSentence,exampleSentenceExtended,partOfSpeech
```

- 三个语境字段均要求非空，分别限制为最多 2,000 字符。
- 保留 RFC4180 引号、换行、UTF-8、500 行和 256 KiB 边界。
- 导入工作区明确提示“短语 → 基础句 → 扩展句”。
- 修正练习编辑器原先按题型硬编码 S1/S2 的问题，编辑 v2 题目时保留原阶段身份。

### 3.3 构建与学习任务

- ContentBuilder 按词库版本选择 v1 或 v2 阶段映射。
- v2 的 S0、S3、S4/S5 分别读取短语、基础句、扩展句。
- 题目构建、覆盖检查、批准、发布和任务 schema 同时支持两套合法 S1/S2 配对。
- 学习端继续使用原六类渲染器，不增加新题型或前端调度逻辑。

### 3.4 设计与视觉证据

- 新增交互说明与可运行视觉审阅稿。
- 新增桌面 1440×1000 与手机 375×812 渲染截图。
- 既有管理端词库页面生产图、对照图和状态矩阵随六列导入界面同步更新。

## 5. 验收中发现并关闭的问题

1. 管理端练习编辑器原先把 `recall_word` 固定写回 S1、把 `multiple_choice` 固定写回 S2，会破坏 v2 的阶段身份；已改为保留当前 item.stage，并增加组件回归。
2. UI 无障碍用例仍按旧 S1 输入题操作；已同步为 S1 选择题，375px 定向用例和全套 UI 均通过。
3. 视觉审阅稿初次加载缺少内联 favicon，产生一次 404；已改为内联图标，最终加载无控制台错误。
4. 迁移测试初稿未直接断言合法内容模型之间也不能改写；已增加数据库触发器断言。
5. 双轴代码复核发现编辑、批准和覆盖检查只校验题型结构，没有按内容模型校验阶段映射；已统一接入模型感知校验，并补充编辑、批准和覆盖回归测试。
6. 双轴代码复核发现视觉审阅稿的 S3、S4、S5 初稿提前呈现答案；已分别改为空格、乱序词块和空白输入，S5 进一步限制为输入非空后才能揭示参考句。

## 6. 兼容与风险边界

- 旧发布版本和旧草稿由迁移默认标记为 v1，不要求补写新字段。
- 新版本使用 v2，不能在创建后切换内容模型。
- 学习任务仍复制批准题目的不可变快照；源词数据后续变化不会改写已生成任务。
- 任务 DTO 继续保留历史字段名 `exampleSentence`，避免破坏既有不可变快照；字段在 S0 中承载的语境由内容模型决定。
- API schema 仅放宽 S1/S2 的两组合法配对，其他阶段与题型组合仍拒绝。
- 当前本机原有 Miniflare 持久化状态在普通 `pnpm dev` 启动时报告 `_cf_ALARM` 旧表结构不兼容。本次未删除该本地状态或其中的管理员数据；隔离构建、Cloudflare 检查和整栈均通过，因此该问题不属于本次代码回归，但本机普通开发状态需另行备份并重建。
- 远端 D1 尚未执行 0011 migration；部署新代码前必须先按发布流程应用远端迁移。
