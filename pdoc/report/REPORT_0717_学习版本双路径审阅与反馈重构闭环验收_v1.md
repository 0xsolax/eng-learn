## 2. 验收结论

学习版本已经形成两条审批路径：

1. 默认进入独立管理端审阅，使用学生同款六类题型和相同判题语义逐题体验。
2. 次要动作一次点击全部通过，内部继续按最多 500 项分批，不再逐项勾选。

审阅页支持反馈打回和结构化直接更正。打回后保存当前反馈并阻断批准、禁用与发布；更正后解决反馈、保持草稿、清空旧模拟结果并要求重新体验。审阅接口全部位于 `/api/admin/*`，不调用课程运行时，也不产生学生课程、课时、任务、答题日志或学习状态写入。

本地实现满足计划中的功能、数据约束、并发恢复、迁移兼容、运行态隔离、真实大样本、浏览器和交付文档门槛。远端发布仍未授权，不能把本报告解释为生产上线完成。

## 3. 实施范围

### 3.1 双路径审批

- 版本详情删除逐项 checkbox、全选和“批准所选项目”。
- 新增“进入审阅模式”和“全部通过（N 项）”。
- 全部通过保留 500 项批次上限、部分成功计数、停止后续批次和权威重读。
- 审阅摘要使用服务端派生计数；待重构或禁用项目存在时冻结全部通过。

### 3.2 独立审阅运行器

- 新增可选 itemId 的管理端路由和薄页面。
- `TaskRenderer` 输入收窄为真实渲染字段，不伪造 courseId 或 sessionId。
- 审阅 GET 只返回当前题 prompt，不返回 answer 或学习运行态字段。
- preview/evaluate 复用纯判题函数，不写 D1。
- 本页成功 evaluate 后才启用逐题通过；切题、更正和 revision 变化都会重新关闭。

### 3.3 反馈与更正

- migration 0012 新增独立的一对一当前反馈表和四个触发器。
- 反馈 trim 后限制为 1 至 2,000 字符，重复打回使用 upsert。
- `request_rework` 支持 draft、approved 和 disabled 项回到 `draft + needs_rework`。
- `correct` 要求 prompt 或 answer 实际变化，解决当前反馈但不自动批准。
- 单项批准、批量批准、禁用和发布均在 service 层预查反馈；数据库触发器提供最终失败关闭。

### 3.4 并发与失败恢复

- 所有决定携带 expected content revision。
- 写响应丢失后只有在 revision 恰好等于请求期望值加一，且批准状态、反馈正文或完整内容同时匹配时，才确认本次决定已提交。
- revision 未变化且写入未提交时保留反馈或更正输入，允许原地重试。
- 写请求与权威重读同时失败时保留输入、锁定全部写入口，只允许重新读取。
- 版本被并发发布时立即移除陈旧题目与写入口，返回版本详情，不重放决定。
- 审阅窗口在组合查询前后复核版本 revision 和状态，拒绝题目与 revision 来自不同时刻的撕裂读取。
- 完整练习读取、preview、evaluate、三类审阅决定、原单项/批量批准和禁用均从同一份最新版本快照取题目内容，避免延迟单项读与新 revision 混用。

## 6. 验收中发现并关闭的问题

1. Cloudflare D1 的 `meta.changes` 会包含触发器副作用。直接更正一条练习并由触发器删除一条反馈时会返回 2，原先按“必须精确等于 1”判断会误报 CAS 冲突。已改为只拒绝少于预期的直接写入行数，并增加 repository 回归测试。
2. 打回或更正写请求与权威重读同时失败时，初版会丢失管理员输入。已改为保留输入、锁定导航和全部写动作，并提供显式权威重读。
3. 审阅中版本并发发布时，初版会按普通网络失败继续调用只支持草稿的审阅 GET。已识别 `source_version_immutable`，立即移除陈旧内容并引导返回版本详情。
4. 同 revision 的写前失败与 revision 已变化初版没有区分。已分别保留原地重试输入或关闭旧输入并展示最新内容。
5. 旧管理端 shell 用例仍查找已移除的“先查看”文案。已更新为“进入审阅模式”，全视口回归通过。
6. D1 测试桩和 API route 测试中出现 7 个严格 lint 问题。已改为 schema parse、明确类型和 Error 拒绝原因，最终 ESLint 通过。
7. 708 项整栈证据初稿错误读取了构建响应中不存在的 `exerciseItemCount`。已按真实 `buildCoverageSchema` 改为 `cells.length`，随后 500 + 208 两批真实请求通过。
8. D1 审阅窗口初版分步读取版本、计数和题目，但没有在组合完成后复核 revision。并发编辑可能产生“新 revision + 旧题目”的撕裂窗口，进而批准管理员没有看到的新内容。现已在查询前后读取版本；revision 改变返回 `conflict`，状态变为非草稿返回 `source_version_immutable`，并增加两项 repository 回归测试。
9. 写响应丢失后的初版恢复只比较最终状态、反馈或内容；若其间又发生一次写入，可能把后一次结果误认作本次成功。现已要求权威结果 revision 必须恰好为 `expected + 1`，更正恢复先读取完整项目、再读取内部一致的审阅窗口，并增加批准、打回和更正三类介入 revision 测试。
10. 旧批量批准和禁用路径初版会先读取项目，再读取最新版本 revision；两次读取间若发生编辑，旧 prompt/answer 可能携带最新 revision 被回写。现已用最新版本快照重新映射目标项目后再写入，并增加批量批准和禁用的并发覆盖测试。
11. 双失败进入恢复锁后，手动“重新读取服务端状态”初版没有识别并发发布返回的 `source_version_immutable`，会继续保留陈旧可变内容。现已复用不可变状态处理，立即清空题目、反馈和更正表单并返回版本详情；新增组件红绿测试。
12. 审阅服务初版使用第一次单项定位读取得到的项目，并只从后续版本快照取得 revision。repository 单项读若延迟，可能把旧 prompt 与当前 revision 组合。现已将完整练习读取、preview、evaluate 和三类审阅决定统一重新映射到当前版本快照项目，并增加延迟定位读回归测试。
