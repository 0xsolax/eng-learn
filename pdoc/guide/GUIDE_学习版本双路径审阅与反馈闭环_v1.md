## 一、功能定位

- 本功能负责草稿学习版本从“练习已构建”到“满足发布条件”的内容质检闭环。 - v1.0.0
- 管理员可以选择逐题真实审阅，也可以一次点击批准当前全部可批准草稿；两条路径共享同一套服务端内容状态和并发规则。 - v1.0.0
- 审阅模式复用学生六类题型的渲染与纯判题语义，但不进入学生课程运行时，不创建课程、课时、任务快照、答题日志或学习状态。 - v1.0.0
- “打回重构”只表示回到可编辑草稿并保存当前反馈，不会自动重新生成或覆盖练习。 - v1.0.0

## 二、入口与相关文件

- 版本详情入口：`src/pages/admin/SourceVersionDetailPage.vue`，提供“进入审阅模式”和“全部通过（N 项）”。 - v1.0.0
- 审阅路由：`/admin/source-versions/:versionId/review/:itemId?`，路由页面为 `src/pages/admin/ExerciseReviewPage.vue`。 - v1.0.0
- 审阅交互协调器：`src/features/admin-content/ExerciseReviewRunner.vue`。 - v1.0.0
- 六类题型渲染：`src/components/task-renderers/TaskRenderer.vue`，输入契约为只包含真实渲染字段的 `TaskRenderDto`。 - v1.0.0
- 管理端 API 客户端：`src/api/adminApi.ts`；所有审阅接口都位于 `/api/admin/*`。 - v1.0.0
- 后端业务规则：`server/services/ContentBuilder.ts`；D1 访问只位于 repository。 - v1.0.0
- 当前反馈迁移：`migrations/0012_add_exercise_review_feedback.sql`。 - v1.0.0

## 三、业务流程

### 3.1 逐题审阅

- 管理员从版本详情进入审阅模式，系统按单词导入顺序和 S0 至 S5 顺序读取一道题。 - v1.0.0
- 审阅 GET 只返回当前题目的 prompt，不返回 answer、课程或课时字段。 - v1.0.0
- 管理员使用学生同款交互完成模拟作答；成功判题后，当前题才允许执行“通过并下一题”。 - v1.0.0
- 模拟答案、得分和预览只保存在当前页面内，刷新后需要重新体验。 - v1.0.0

### 3.2 全部通过

- “全部通过”是一轮用户点击，不是新增的无限大服务端事务。 - v1.0.0
- 前端继续调用现有批量批准接口，每批最多 500 项；超过 500 项时按稳定顺序分批。 - v1.0.0
- 任一批次失败后立即停止后续批次，显示已确认批准数量并权威重读服务端状态。 - v1.0.0
- 存在待重构反馈、禁用项目或覆盖缺口时，全部通过必须冻结。 - v1.0.0

### 3.3 反馈与直接更正

- “打回重构”要求填写 trim 后 1 至 2,000 字符的反馈；成功后项目变为 `draft + needs_rework`。 - v1.0.0
- 每道题只保存一条当前未处理反馈；再次打回会更新该反馈，不保存历史评论线程。 - v1.0.0
- “直接更正”读取完整结构化练习，允许修改 prompt 或 answer，但不允许改变 stage 或 task type。 - v1.0.0
- 更正必须造成实际内容变化；保存后反馈被解决、项目保持 draft、旧模拟结果清空，必须重新体验后才能批准。 - v1.0.0
- 草稿版本内的 approved 或 disabled 项目也可以重新打回或直接更正。 - v1.0.0

## 四、状态与数据不变量

- `ExerciseItemStatus` 始终只有 `draft / approved / disabled`，`needs_rework` 是由 draft 和当前反馈派生的审阅状态。 - v1.0.0
- 当前反馈只能属于 draft 项目；数据库触发器禁止 `approved/disabled + 当前反馈`。 - v1.0.0
- 内容实际改变时，数据库触发器删除当前反馈；未改变内容不能伪装成已更正。 - v1.0.0
- 带反馈项目不能通过单项批准、批量批准、禁用或发布绕过。 - v1.0.0
- 每次成功的批准、打回或更正只增加一次 `source_versions.content_revision`。 - v1.0.0
- 已发布版本不可原地反馈、更正或重新批准；后续修改必须创建下一草稿版本。 - v1.0.0
- 已有课程和 `lesson_tasks` 快照不被草稿审阅回写。 - v1.0.0

## 五、接口与分层边界

- 审阅读取、preview、evaluate 和 decision 都是管理端接口，不得新增或复用 `/api/app/*` 写路径。 - v1.0.0
- route 只负责解析、schema 校验和调用 `ContentBuilder`；审批、反馈和发布门禁必须留在 service。 - v1.0.0
- preview 和 evaluate 复用纯 `evaluateTaskSubmission` 语义，不调用 `CourseRuntime`。 - v1.0.0
- `LessonRunner` 不得嵌入管理端；管理端只复用纯题型渲染组件。 - v1.0.0
- D1 SQL 和反馈触发器只由 repository 与 migration 管理，Vue 和 service 不直接拼 SQL。 - v1.0.0

## 六、并发与失败恢复

- 所有审阅决定必须携带管理员打开题目时的 expected content revision。 - v1.0.0
- 审阅窗口在组合查询前后复核版本状态和 revision；两次不一致时返回冲突或不可变错误。 - v1.0.0
- 完整练习读取、preview、evaluate、审阅决定、原单项/批量批准和禁用都从当前版本快照重新映射目标项目，不能把延迟单项读与新 revision 混用。 - v1.0.0
- 写响应丢失后，只有权威 revision 恰好为 `expected + 1` 且结果匹配时，才能确认本次决定已提交。 - v1.0.0
- revision 未变化时保留反馈或更正输入并允许原地重试；revision 已变化时关闭旧输入，要求基于最新内容重新决定。 - v1.0.0
- 写请求和权威重读同时失败时保留输入、锁定全部写入口，只允许手动重新读取服务端状态。 - v1.0.0
- 重新读取发现版本已发布或归档时，立即清除陈旧题目与可变表单并返回版本详情。 - v1.0.0
- 任何失败恢复都不得自动使用新 revision 重放管理员基于旧内容作出的决定。 - v1.0.0

## 七、响应式与交互边界

- 479px 及以下为只读边界，不渲染可提交题目、反馈、更正或批准入口。 - v1.0.0
- 480px 起恢复完整审阅交互。 - v1.0.0
- 反馈面板支持键盘聚焦和 Escape 关闭，关闭后焦点返回触发按钮。 - v1.0.0
- 写操作期间禁用重复触发；操作结果通过 status 或 alert 区域反馈，不能只依赖颜色。 - v1.0.0

## 八、测试与发布门禁

- 必测层级包括共享 schema、service、内存 repository、D1 repository、API 认证与 Origin、Vue 组件、隔离整栈和真实浏览器。 - v1.0.0
- 整栈验收必须比较 `courses`、`lesson_sessions`、`lesson_tasks`、`review_logs`、`user_word_states` 在审阅前后的完整结果，证明学生运行态零污染。 - v1.0.0
- 大样本必须覆盖超过 500 项的真实分批路径；当前验收样本为 708 项，批次为 500 + 208。 - v1.0.0
- 发布前必须通过 typecheck、lint、全量测试、生产构建、敏感制品扫描、Cloudflare 启动/类型检查和 `git diff --check`。 - v1.0.0
- 远端 D1 必须先应用 migration 0012，再部署使用反馈表的新 Worker；未获明确授权不得执行远端 migration、部署或生产写入。 - v1.0.0
- 代码回滚时保留 migration 0012；旧 Worker 可继续读取原表，带反馈项目通过触发器安全失败关闭。 - v1.0.0

## 九、当前限制

- 第一版不记录多轮反馈历史、审核人、多人会签、通知或审阅会话。 - v1.0.0
- 第一版不自动生成被打回项目，不持久化管理员模拟答案或得分。 - v1.0.0
- 第一版不根据模拟得分自动批准，批准始终需要管理员明确操作。 - v1.0.0
- 若未来需要反馈审计，应另建 append-only 审阅事件模型，不在当前反馈表继续堆字段。 - v1.0.0
